Threat Hunting with Sigma Rules — Writing Detections That Actually Work

2026-06-16 · 1 min read · Security · threat-hunting sigma detection siem blue-team edr soc mitre-attack

Sigma is the vendor-neutral standard for writing detection rules that compile to any SIEM. Covers the Sigma format in depth, writing rules for real ATT&CK techniques, converting to Splunk/Elastic/QRadar queries, hunting methodologies, and building a detection pipeline that keeps pace with evolving adversaries.

Threat hunting is not waiting for an alert. It is the proactive search for adversary activity that your current detections have missed — the assumption that a breach has occurred or is occurring, and a structured effort to find it.

Sigma is the language that makes detections portable. Write a rule once, compile it to Splunk, Elastic, Azure Sentinel, QRadar, or Wazuh. The Sigma community maintains a library of thousands of rules mapped to MITRE ATT&CK — the best starting point for any detection programme. This post covers the format, the methodology, and the craft of writing rules that catch real adversary behaviour without flooding analysts with noise.

01 Reactive Monitoring vs Proactive Hunting

Monitoring (Reactive)Threat Hunting (Proactive)
Waits for an alert to fireAssumes the alert will not fire — actively looks for what signatures miss
Driven by vendor signatures and correlation rulesDriven by threat intelligence, ATT&CK techniques, and analyst hypotheses
Optimised for speed of responseOptimised for depth of investigation
Alert → Triage → EscalateHypothesis → Evidence search → Either: nothing found OR new detection rule created
Effective against known-badEffective against novel techniques and long-dwell-time attackers

The output of a threat hunt is either confirmation that a technique is absent (and therefore implicitly covered) or a new detection rule for the thing you found. Every hunt should produce a Sigma rule, even if the hunt found nothing — you have now validated that the absence of that artefact means the technique was not used.

02 The Sigma Format

A Sigma rule is a YAML document describing a detection. It is structured into a header (metadata), a detection section (the logic), and conditions (how to combine the logic).

yaml 
# Sigma rule skeleton — every rule follows this structure
title: Descriptive Name of What This Detects        # required
id: 8b6b0def-1234-5678-9abc-def012345678            # UUID, required, generate with uuidgen
status: experimental                                 # stable | test | experimental | deprecated
description: |                                       # required
    One paragraph explaining what behaviour this detects,
    why it matters, and what adversary technique it covers.
references:                                          # optional but important
    - https://attack.mitre.org/techniques/T1059/001/
    - https://example.com/blog/technique-analysis
author: Your Name                                    # optional
date: 2026-06-16                                    # optional
tags:                                               # MITRE ATT&CK mapping
    - attack.execution
    - attack.t1059.001
logsource:                                          # what log source
    category: process_creation                      # Sigma category (maps to EventID etc.)
    product: windows                                # product filter
detection:                                          # the logic lives here
    selection_main:                                 # named selection block
        Image|endswith: '\powershell.exe'
        CommandLine|contains|all:                   # ALL of these must match
            - '-enc'
            - '-NonI'
    selection_alt:
        Image|endswith: '\pwsh.exe'
        CommandLine|contains: '-enc'
    filter_legitimate:                              # whitelist / false positive reduction
        ParentImage|endswith:
            - '\msiexec.exe'
            - '\wmiprvse.exe'
    condition: (selection_main or selection_alt) and not filter_legitimate
falsepositives:                                     # document known FPs
    - Legitimate software deployment tools that use encoded commands
level: medium                                       # informational | low | medium | high | critical

03 Detection Logic — Field Modifiers

Sigma's power is in its field modifiers. They translate to native SIEM query language when compiled.

ModifierMeaningExample
`contains`Field contains the value as a substring`CommandLine|contains: '-enc'`
`startswith`Field starts with the value`Image|startswith: 'C:\\Windows\\'`
`endswith`Field ends with the value`Image|endswith: '\\powershell.exe'`
`contains|all`Field contains ALL listed values`CommandLine|contains|all: ['-enc', '-w', 'hidden']`
`re`Regular expression match`CommandLine|re: '(?i)iex\\s*\\('`
`cidr`IP matches CIDR range`DestinationIp|cidr: '10.0.0.0/8'`
`base64offset|contains`Value appears as base64 in field (any offset)`CommandLine|base64offset|contains: '/bin/sh'`
`windash`Match both `-flag` and `/flag` variants`CommandLine|windash|contains: '-enc'`
`gt`, `lt`, `gte`, `lte`Numeric comparison`count(CommandLine)|gt: 50`

04 Real Detection Rules — ATT&CK Technique by Technique

T1059.001 — PowerShell with encoded commands

yaml 
title: PowerShell Encoded Command Execution
id: c539afac-c12a-46ed-b1c5-924a97ec9de8
status: stable
description: |
    Detects PowerShell execution with the -EncodedCommand flag, which is used
    to pass base64-encoded scripts. While occasionally legitimate (some deployment
    tools use it), the combination with -NonInteractive and -WindowStyle Hidden
    is almost always malicious — it suppresses the console window and disables
    user interaction, characteristic of stager and dropper behaviour.
references:
    - https://attack.mitre.org/techniques/T1059/001/
tags:
    - attack.execution
    - attack.t1059.001
logsource:
    category: process_creation
    product: windows
detection:
    selection:
        Image|endswith:
            - '\powershell.exe'
            - '\pwsh.exe'
        CommandLine|windash|contains|all:
            - '-enc'          # -EncodedCommand, -Enc, -en, -e (PowerShell accepts prefixes)
            - '-w'
            - 'hidden'        # -WindowStyle Hidden — suppress console window
    condition: selection
falsepositives:
    - Rare: some enterprise deployment tools use encoded commands with hidden windows
    - Legitimate Citrix/remote desktop software has been observed using this pattern
level: high

# ─────────────────────────────────────────────────────────────────
# WHY THIS WORKS: base64-encoded payload + hidden window = attacker
# trying to run code without a visible terminal. The combination is
# the signal — individual flags appear in legit software but combined
# with a hidden window, this is ~95% malicious in practice.
# ─────────────────────────────────────────────────────────────────

T1003.001 — LSASS memory dump

yaml 
title: LSASS Memory Access by Unusual Process
id: a0184def-a2b9-40e7-88f5-31a9734b65a3
status: stable
description: |
    Detects process access to LSASS (Local Security Authority Subsystem Service)
    memory by processes that are not legitimate security or system processes.
    LSASS holds credential material (NTLM hashes, Kerberos tickets) in memory.
    Access by unusual processes indicates credential dumping — Mimikatz,
    procdump, or custom tools reading the LSASS process memory space.
references:
    - https://attack.mitre.org/techniques/T1003/001/
tags:
    - attack.credential_access
    - attack.t1003.001
logsource:
    category: process_access          # Sysmon Event ID 10
    product: windows
detection:
    selection:
        TargetImage|endswith: '\lsass.exe'
        GrantedAccess|contains:       # access rights that allow memory read
            - '0x1010'                # PROCESS_VM_READ + PROCESS_QUERY_INFORMATION
            - '0x1410'
            - '0x147a'
            - '0x143a'
            - '0x1438'
            - '0x1fffff'              # PROCESS_ALL_ACCESS — very suspicious
    filter_legitimate:
        SourceImage|endswith:
            - '\MsMpEng.exe'          # Windows Defender
            - '\svchost.exe'
            - '\csrss.exe'
            - '\wininit.exe'
            - '\wmiprvse.exe'
            - '\taskmgr.exe'          # Task Manager (legitimate LSASS access)
            - '\lsm.exe'
    condition: selection and not filter_legitimate
falsepositives:
    - AV and EDR products legitimately access LSASS — add to filter
    - Some forensic/backup tools access process memory
level: critical

T1558.003 — Kerberoasting

yaml 
title: Kerberoasting — Service Ticket Request with RC4 Encryption
id: b3ae2f78-d4d7-4e93-a8d4-23b71bd3b7e1
status: stable
description: |
    Detects Kerberoasting: the technique of requesting Kerberos service tickets
    for accounts with SPNs (Service Principal Names) and cracking them offline.
    The tell is RC4 encryption (etype 0x17 / 23) for the service ticket —
    modern environments use AES (etype 0x11/0x12). An attacker requests RC4
    tickets because they are faster to crack.
    
    Key signal: multiple RC4-encrypted TGS requests for different service accounts
    from a single source within a short window (automated enumeration).
references:
    - https://attack.mitre.org/techniques/T1558/003/
    - https://adsecurity.org/?p=2293
tags:
    - attack.credential_access
    - attack.t1558.003
logsource:
    product: windows
    service: security           # Windows Security Event Log
detection:
    selection:
        EventID: 4769           # Kerberos Service Ticket Operations
        TicketOptions: '0x40810000'
        TicketEncryptionType: '0x17'   # RC4-HMAC — the Kerberoasting fingerprint
    filter_computer_accounts:
        ServiceName|endswith: '$'  # machine accounts ending in $, not user accounts
    condition: selection and not filter_computer_accounts
falsepositives:
    - Legacy systems (pre-Windows 2008) that do not support AES Kerberos
    - Applications explicitly configured to use RC4 for compatibility
level: high

# ─────────────────────────────────────────────────────────────────
# TUNING NOTE: Aggregate this rule. One RC4 TGS = possible legacy
# system. Five RC4 TGS requests for different accounts within 60
# seconds from the same IP = Kerberoasting. Use statistical detection
# or temporal aggregation in your SIEM to raise severity on volume.
# ─────────────────────────────────────────────────────────────────

T1021.002 — PsExec-style lateral movement

yaml 
title: Remote Service Installation via ADMIN$ Share (PsExec Pattern)
id: f23a2a60-6ef1-4bb2-9b15-8d34e8d7b5d0
status: stable
description: |
    Detects the PsExec lateral movement pattern: copying a binary to the ADMIN$
    share of a remote host and then creating a service to execute it. This is the
    technique used by PsExec (Sysinternals), Impacket psexec.py, and many C2
    frameworks for lateral movement. Key artefacts:
    1. File written to C:\Windows\PSEXESVC.exe (or random name)
    2. Service created with that binary path
    3. Source is typically a remote SMB connection (logon type 3)
references:
    - https://attack.mitre.org/techniques/T1021/002/
tags:
    - attack.lateral_movement
    - attack.t1021.002
logsource:
    product: windows
    service: system
detection:
    selection_service_install:
        EventID: 7045               # New service installed
        ServiceFileName|contains:
            - 'PSEXESVC'
            - 'PAEXEC'
    selection_generic_suspicious:
        EventID: 7045
        ServiceFileName|startswith:
            - 'C:\Windows\Temp\'
            - 'C:\Users\'
            - 'C:\ProgramData\'
    condition: selection_service_install or selection_generic_suspicious
falsepositives:
    - Authorised remote administration using PsExec during maintenance
    - SCCM and similar management platforms installing services remotely
level: high

T1048 — DNS exfiltration detection

yaml 
title: Potential DNS Tunneling or Exfiltration
id: a7b3c4d5-e6f7-8901-abcd-ef0123456789
status: experimental
description: |
    Detects potential DNS-based data exfiltration by identifying DNS queries
    with anomalously long subdomain labels (>50 characters). Legitimate DNS
    queries rarely have subdomains longer than 30 characters. Attackers use
    DNS tunneling tools (iodine, dnscat2, DNSExfiltrator) to encode data
    in subdomain labels of queries to attacker-controlled nameservers.
references:
    - https://attack.mitre.org/techniques/T1048/
    - https://github.com/iagox86/dnscat2
tags:
    - attack.exfiltration
    - attack.t1048
logsource:
    category: dns
detection:
    selection:
        query|re: '([a-zA-Z0-9]{50,})\.'
    filter_known_long:
        query|endswith:
            - '.in-addr.arpa'
            - '.ip6.arpa'
    filter_cdn:
        query|endswith:
            - '.cloudfront.net'
            - '.azureedge.net'
            - '.akamaiedge.net'
    condition: selection and not 1 of filter_*
falsepositives:
    - Some legitimate CDN and cloud services use very long subdomains
    - Telemetry and analytics services sometimes encode data in subdomains
    - Adjust the 50-character threshold based on your environment baseline
level: medium

05 Converting Sigma Rules to SIEM Queries

bash 
# Install sigma-cli (official converter)
pip3 install sigma-cli
# Install backend plugins for your SIEM
pip3 install pySigma-backend-splunk
pip3 install pySigma-backend-elasticsearch
pip3 install pySigma-backend-qradar
pip3 install pySigma-backend-azure-monitor

# Convert a single rule to Splunk
sigma convert -t splunk rules/my_rule.yml

# Convert with specific pipeline (field name mapping for your SIEM)
sigma convert -t splunk -p splunk_windows rules/my_rule.yml

# Convert entire directory
sigma convert -t splunk -p splunk_windows rules/ -o splunk_queries.txt

# Convert to Elastic (ECS fields)
sigma convert -t elasticsearch -p ecs_windows rules/ -f ndjson > rules.ndjson

# Convert to Wazuh (custom Wazuh backend)
sigma convert -t wazuh rules/lsass_access.yml

# Convert to KQL (Azure Sentinel / Microsoft Defender)
sigma convert -t kusto -p azure_windows rules/kerberoasting.yml

Example — Splunk query output for the PowerShell rule

text 
# Generated by: sigma convert -t splunk -p splunk_windows powershell_enc.yml

index=windows source="XmlWinEventLog:Microsoft-Windows-Sysmon/Operational" EventCode=1
( Image IN ("*\\powershell.exe", "*\\pwsh.exe") )
CommandLine IN ("*-enc*", "*-Enc*", "*/enc*")
CommandLine IN ("*-w*", "*-W*", "/w*")
CommandLine IN ("*hidden*", "*Hidden*", "*HIDDEN*")
| eval risk_score=80, technique="T1059.001"
| table _time, host, user, Image, CommandLine, ParentImage, risk_score

Example — Elasticsearch/Kibana KQL

text 
# Generated by: sigma convert -t elasticsearch -p ecs_windows powershell_enc.yml

process.executable: (*\powershell.exe OR *\pwsh.exe) AND
process.command_line: (*-enc* OR *-Enc* OR */enc*) AND
process.command_line: (*-w* OR */w*) AND
process.command_line: (*hidden* OR *Hidden*)

06 Hunting Methodology — The Hypothesis Framework

Unstructured hunting is random. The hypothesis framework gives each hunt a defined scope, expected output, and success criteria.

text 
HUNT HYPOTHESIS TEMPLATE
─────────────────────────────────────────────────────────────────
Hypothesis:
  'An adversary with valid credentials has used WinRM/PSRemoting
   to move laterally between workstations, which would not be
   detected by our current network-based detection'

Based On:
  - Recent threat intel: APT group targeting finance sector via WinRM
  - ATT&CK T1021.006 not covered in our detection matrix
  - No current alert rule for workstation-to-workstation WinRM

Data Sources Required:
  - Windows Security Event Log: 4624 (logon), 4103/4104 (PowerShell)
  - Sysmon: network connections from wsmprovhost.exe
  - Firewall logs: TCP/5985 (HTTP WinRM), TCP/5986 (HTTPS WinRM)

Query Strategy:
  1. Find all 4624 logon events with LogonType=3 (network) from
     source hosts that are workstations (not servers)
  2. Correlate with wsmprovhost.exe process creation on target host
  3. Flag workstation-to-workstation lateral movement

Success Criteria (either outcome is success):
  A) Found evidence → escalate as incident + write detection rule
  B) No evidence found → document as 'technique not observed in
     this period' + write Sigma rule to detect it going forward

Time Box: 2 hours
─────────────────────────────────────────────────────────────────

Hunt query — WinRM lateral movement

yaml 
# Sigma rule written DURING the hunt — captures the hypothesis as a detection
title: WinRM Lateral Movement Between Workstations
id: c7b2a1e0-d3f4-5678-90ab-cdef01234567
status: experimental
description: |
    Detects WinRM/PowerShell Remoting usage between workstations, which is rarely
    legitimate in most environments. Legitimate WinRM use is typically from
    management servers or jump hosts to workstations/servers — not between peers.
    Workstation-to-workstation WinRM is a strong indicator of lateral movement.
tags:
    - attack.lateral_movement
    - attack.t1021.006
logsource:
    product: windows
    service: security
detection:
    selection_winrm_network_logon:
        EventID: 4624
        LogonType: 3           # Network logon
        AuthenticationPackageName: 'Kerberos'
        # ProcessName will be wsmprovhost.exe for WinRM sessions on target
    selection_process_on_target:
        EventID: 4688          # Process creation
        NewProcessName|endswith: '\wsmprovhost.exe'
    filter_servers:
        ComputerName|contains:
            - 'SRV-'
            - 'SERVER'
            - 'DC-'
    condition: (selection_winrm_network_logon or selection_process_on_target)
              and not filter_servers
falsepositives:
    - IT administrators remoting into workstations for support
    - Management tooling configured to use WinRM against workstations
level: medium

07 Sigma Rule Quality Checklist

Before merging a detection rule to your production ruleset:

✓ Mapped to one or more ATT&CK technique IDs in the tags field
✓ id field contains a unique UUID (uuidgen to generate)
✓ description explains WHY this is suspicious, not just what it matches
✓ False positives documented — every known exception is listed
✓ Filter blocks exclude legitimate uses (don't just have selection blocks)
✓ Tested against known-good baseline: zero false positives in 24-hour dry run
✓ Tested against a known attack: Atomic Red Team test fires the rule
✓ level reflects actual risk: most rules should be medium — use critical sparingly
✓ Peer-reviewed by a second analyst before production deployment
✗ Do not ship a rule without testing it — untested rules erode trust in your alerting
✗ Do not copy rules from the SigmaHQ community repository without testing them in your environment first — your baseline is different

The 80/20 rule for detection coverage

You cannot write rules for every ATT&CK technique simultaneously. Prioritise by: (1) techniques in the CISA KEV feed — actively exploited right now, (2) techniques in threat reports targeting your sector, (3) techniques you observed in your last red team engagement. Twenty well-tuned, low-noise rules covering your actual threat landscape beat two hundred noisy rules nobody investigates.
