Purple Team Exercise — Planning and Running a Full Scenario

2026-06-17 · 2 min read · Security · purple-team detection-engineering atomic-red-team mitre-attack

A step-by-step guide to running a structured purple team exercise: choosing a threat scenario that matches real risk, defining success criteria up front, executing the kill chain in coordination with the blue team, and converting the results into a lasting detection programme.

A purple team exercise isn't a pentest with a defender in the room, and it isn't a tabletop. It's a structured, collaborative test of detection and response capability against specific, agreed techniques — with the explicit goal of leaving the defensive programme measurably better than it started. That goal shapes every decision below.

Step 1 — Choosing a Threat Scenario

The scenario should be the threat actually relevant to the organisation, not the most interesting one to run. Three sources, in priority order:

1. Threat intelligence specific to the organisation's sector. If ransomware affiliates targeting your industry are documented as exploiting backup infrastructure or harvesting infostealer logs for initial access, that's the scenario — not because it's novel, but because it's what's actually being used against peers.

2. Gaps surfaced by a recent incident or near-miss, internal or industry-wide. A purple team exercise that re-creates a recent real incident under controlled conditions validates whether the fix actually closed the gap, which a post-incident report alone can't prove.

3. Known coverage gaps from prior exercises or assessments. If the last purple team round found that lateral movement detection was weak, this round's scenario should specifically exercise lateral movement techniques again, post-remediation — not move on to an unrelated kill chain stage.

text Example scenario definition 
Scenario: "Ransomware affiliate, initial access via phished credentials,
           targeting backup infrastructure before encryption"

Kill chain stages in scope:
  1. Initial access      -- T1566.001 (spearphishing attachment)
  2. Execution            -- T1059.001 (PowerShell)
  3. Persistence          -- T1547.001 (registry run key)
  4. Credential access     -- T1003.001 (LSASS memory)
  5. Lateral movement      -- T1021.002 (SMB/admin shares)
  6. Discovery             -- T1018 (remote system discovery, targeting backup infra)
  7. Impact (simulated only) -- T1490 (inhibit system recovery -- backup deletion)

Explicitly out of scope: actual encryption, actual data exfiltration

Step 2 — Defining Success Criteria Before Execution

Agree on what "detected" and "responded" mean, in writing, before the first technique runs — not in the after-action review when it's convenient to redefine them. Vague criteria produce a debate about scoring instead of a measurement of capability.

CriterionBad definitionGood definition
Detected"The SOC noticed something""An alert fired in the SIEM, mapped to the specific technique executed, within an agreed time window (e.g. 15 minutes)"
Responded"Someone looked at it""A documented triage action was taken — ticket created, analyst assigned, and an investigative action logged — within the agreed SLA for that alert's severity"
Contained"It would have been stopped eventually""A specific containment action (isolate host, disable account, block IOC) was executed, and the red team's follow-on action was demonstrably blocked by it"
Score per technique, not per exercise

A single pass/fail for the whole exercise hides where the actual gap is. Score detected/responded/contained independently for every technique in the kill chain — this is what makes the after-action report actionable instead of a single disappointing or falsely reassuring grade.

Step 3 — Executing the Kill Chain in Coordination

Purple team execution differs from a red team engagement specifically in the coordination model: the blue team knows an exercise is happening (even if not the exact techniques or timing), and there's a live communication channel open throughout — typically a dedicated chat channel — so the red team can confirm "technique X just executed" immediately after each step, enabling real-time correlation against what the blue team's tooling actually caught.

bash Executing scenario techniques with Atomic Red Team, technique by technique 
# Each technique run is logged with a timestamp posted to the shared channel
# immediately before execution, enabling precise blue-team correlation

Invoke-AtomicTest T1566.001 -ShowDetailsBrief
Invoke-AtomicTest T1566.001

Invoke-AtomicTest T1059.001
Invoke-AtomicTest T1547.001
Invoke-AtomicTest T1003.001
Invoke-AtomicTest T1021.002
Invoke-AtomicTest T1018

After each technique executes, the blue team independently checks their tooling (SIEM, EDR console) for a corresponding alert and records, against the success criteria from Step 2: did it fire, how long did it take, and what action followed. This happens live, technique by technique — not as a single retrospective log review at the end, which loses the timing data that's often the most useful signal.

Step 4 — Converting Findings Into a Detection Programme

The exercise itself is not the deliverable — the detection improvements that outlive it are. For every technique that scored "not detected" or "detected but not responded to within SLA," the output is a concrete artifact, not a recommendation to "improve monitoring":

Gap foundConcrete artifact produced
T1547.001 (registry run key persistence) — not detectedNew Sigma rule targeting the specific persistence pattern observed, validated by re-running the same atomic test against it before closing the gap
T1003.001 (LSASS memory access) — detected, but response exceeded SLAUpdated SOAR playbook auto-isolating the host and disabling the account on this specific alert category, removing the manual triage delay
T1018 (remote system discovery targeting backup infra) — not detectedNew detection rule plus a network segmentation finding escalated separately — backup infrastructure should not have been reachable for this discovery technique to succeed at all
Close the loop — re-test the same techniques next cycle

The single most common purple-team failure mode is treating each exercise as a standalone event. Schedule the next purple team round to specifically re-run the techniques that scored poorly this time, first, before introducing new scenario coverage — proving the new Sigma rule and SOAR playbook actually work against a live re-run is the only way to confirm the gap is closed rather than just documented.

Map results to an ATT&CK Navigator layer

Maintain a living Navigator layer scored by detection confidence per technique (the same approach covered in the companion notes on reading malware reports), updated after every purple team cycle. Over several quarters this becomes the most honest picture of actual detection coverage the organisation has — far more reliable than a tool vendor's claimed coverage matrix, because every cell is backed by a real, timestamped test.
