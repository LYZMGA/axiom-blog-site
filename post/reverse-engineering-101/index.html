Reverse Engineering 101 — Reading What the Compiler Wrote

2026-06-16 · 1 min read · Learn · reverse-engineering malware-analysis assembly ghidra ida x64dbg binary-analysis exploit-dev

Reverse engineering demystified. Covers static vs dynamic analysis, essential tools, x86/x64 assembly fundamentals, reading decompiler output, patching binaries, unpacking, and a worked malware analysis walkthrough — from running sample to extracted C2 address.

Reverse engineering is reading a program that has no source code. Every compiled binary is a message the compiler left — assembly instructions, memory layout, data structures, control flow. Learn to read it and you can understand what any program does: malware, closed-source software, firmware, drivers, proprietary protocols.

For security practitioners, RE is the skill that separates operators who can analyse what they find from operators who have to rely on someone else's write-up. You cannot fully defend against what you cannot understand. You cannot responsibly exploit what you have not analysed. This guide builds the foundation from first principles.

01 Static vs Dynamic Analysis

ApproachWhat It MeansToolsWhen to Use
**Static analysis**Examine the binary without running it. Read disassembly, decompiled code, strings, imports, headersGhidra, IDA Pro/Free, Binary Ninja, Cutter (r2 GUI), strings, file, pestoolFirst pass on unknown samples. Safe — no execution. Slower for packed/obfuscated samples
**Dynamic analysis**Run the binary in a controlled environment and observe behaviour: API calls, network traffic, file writes, registry changesx64dbg, OllyDbg, WinDbg, API Monitor, Procmon, Wireshark, Cuckoo SandboxAfter static gives you context. Faster for packers. Requires isolated environment (VM snapshots)
**Hybrid**Alternate between both — static to understand structure, dynamic to confirm behaviour, back to static to understand whyAll of the aboveComplex malware analysis, exploit development, crackmes
Always analyse in an isolated environment

Never run unknown malware on your host machine or any machine connected to a production network. Use a dedicated VM with network isolation — block all outbound except to your own lab DNS/C2 catcher (INetSim, FakeNet-NG). Revert to a clean snapshot before each analysis session. Some malware detects VMs and behaves differently — disable VMware guest additions and common VM artefacts.

03 x86/x64 Assembly Fundamentals

You do not need to memorise the entire x86 instruction set. You need to recognise the 30 instructions that appear in 90% of code. The rest you look up.

Data movement and arithmetic

asm 
; MOV — move data (most common instruction)
mov eax, 0x10         ; eax = 0x10 (immediate value)
mov eax, [ebp-4]      ; eax = value at memory address (ebp - 4)
mov [esp+8], eax      ; write eax to memory at (esp + 8)

; LEA — load effective address (compute pointer arithmetic)
lea eax, [ebp-0x50]   ; eax = (ebp - 0x50) — address, not value
                      ; commonly used for: getting address of local variable
                      ;                    fast multiply: lea eax,[eax+eax*4] = eax*5

; Arithmetic
add eax, 4            ; eax = eax + 4
sub esp, 0x20         ; esp = esp - 32  (allocate stack space in function prologue)
imul eax, ecx, 3      ; eax = ecx * 3
inc ecx               ; ecx++
dec ecx               ; ecx--

; Bitwise
xor eax, eax          ; eax = 0  (common idiom — faster than mov eax, 0)
and eax, 0xFF         ; mask to lowest byte
or  eax, 0x100        ; set bit 8
shl eax, 2            ; eax <<= 2  (multiply by 4)
shr eax, 1            ; eax >>= 1  (divide by 2)

Control flow

asm 
; CMP — compare (sets EFLAGS, does not store result)
cmp eax, 0            ; eax - 0, set ZF if equal
cmp eax, ebx          ; eax - ebx

; Conditional jumps (read the condition, not just 'jump')
jz  .label            ; jump if ZF=1 (zero / equal)
jnz .label            ; jump if ZF=0 (not zero / not equal)
jl  .label            ; jump if less (signed)
jg  .label            ; jump if greater (signed)
jle .label            ; jump if less or equal
jge .label            ; jump if greater or equal
jb  .label            ; jump if below (unsigned)
ja  .label            ; jump if above (unsigned)

; TEST — bitwise AND, only sets flags
test eax, eax         ; if eax == 0: ZF=1
jz .null_check        ; common null pointer check pattern

; Unconditional
jmp .label            ; always jump
call function         ; push return address, jump to function
ret                   ; pop return address into EIP/RIP, jump

; The C → ASM translation of an if statement:
; if (a == b) { do_something(); }
;  cmp eax, ebx
;  jnz .skip          ; if NOT equal, skip the body
;  call do_something
; .skip:

Stack operations and function calls

asm 
; Standard x86 function prologue
push ebp              ; save caller's base pointer
mov  ebp, esp         ; establish new stack frame
sub  esp, 0x28        ; allocate 40 bytes for local variables

; Standard epilogue
mov  esp, ebp         ; (or: leave — same thing)
pop  ebp
ret

; x64 calling convention (Windows): RCX, RDX, R8, R9, then stack
; Arguments:  1st=rcx  2nd=rdx  3rd=r8  4th=r9
; Return value: rax
; x64 calling convention (Linux SysV): RDI, RSI, RDX, RCX, R8, R9

; Example: recognising strcmp in disassembly
; Source: if (strcmp(s1, s2) == 0) { ... }
;
; lea  rcx, [s1]     ; 1st arg: s1
; lea  rdx, [s2]     ; 2nd arg: s2
; call strcmp
; test eax, eax      ; strcmp returns 0 if equal
; jnz  .not_equal

04 Working with Ghidra

Ghidra's decompiler turns disassembly into readable pseudo-C. It is not perfect — names are lost, types are guessed, and complex patterns confuse it — but it dramatically reduces analysis time.

Ghidra workflow

text 
1. New Project → Import file → auto-analysis (accept defaults)
2. Open CodeBrowser → Symbol Tree → Functions → main
3. Decompiler window (right): pseudo-C — rename variables with 'L'
4. Disassembly window (left): assembly — cross-references with 'X'
5. String search: Search > For Strings → click to navigate to usage
6. Rename: right-click symbol → Rename Symbol (persists in project)
7. Retype: right-click variable in decompiler → Retype Variable
8. Cross-references: right-click function → References → Show References
9. XREF usage: see every caller of a function
10. Script Manager: run scripts (Python/Java) for automation

Common decompiler patterns to recognise

c 
/* Ghidra decompiler output — before renaming */
void FUN_00401000(void) {
    char *pcVar1;
    int iVar2;
    undefined local_54 [64];
    
    pcVar1 = (char *)FUN_00405320(local_54, 0x40);  // suspicious
    iVar2 = FUN_0040a0c0(pcVar1, "MSSDK32.dll");    // strcmp-like
    if (iVar2 == 0) {
        FUN_00408910(1);
    }
    FUN_004012a0(local_54, &DAT_00408000, 0x20);
    return;
}

/* After RE work: renamed based on imports + behaviour analysis */
void check_and_load_module(void) {
    char module_name [64];
    int is_match;
    
    /* GetModuleFileNameA writes into buffer */
    GetModuleFileNameA(NULL, module_name, 0x40);
    is_match = strcmp(module_name, "MSSDK32.dll");
    if (is_match == 0) {
        ExitProcess(1);  /* self-terminate if run as a DLL */
    }
    /* XOR decryption of embedded payload */
    xor_decrypt(module_name, &encrypted_blob, 0x20);
    return;
}

05 Binary Patching — Before and After

Patching a binary means modifying its instructions directly — changing a conditional jump to an unconditional one, NOPping out a check, or replacing a hardcoded value. Common use cases: bypassing licence checks in crackmes, neutralising self-destruct routines in malware samples, forcing code paths for analysis.

Example: bypassing a licence check

diff 
; Function: check_licence() at 0x00401200
; Goal: make the function always return 'valid' (1)
; regardless of actual licence input

; BEFORE patch — original disassembly
  00401200: push  ebp
  00401201: mov   ebp, esp
  00401203: call  verify_key_hash        ; returns 0 (invalid) or 1 (valid)
  00401208: test  eax, eax               ; is result zero?
- 0040120A: jz    0x00401230             ; if zero (invalid), jump to fail branch
  0040120C: mov   eax, 1                 ; valid path: return 1
  00401211: pop   ebp
  00401212: ret
  ...
  00401230: mov   eax, 0                 ; fail branch: return 0
  00401235: pop   ebp
  00401236: ret

; AFTER patch — jz changed to jmp (unconditional)
  00401200: push  ebp
  00401201: mov   ebp, esp
  00401203: call  verify_key_hash
  00401208: test  eax, eax
+ 0040120A: jmp   0x0040120C             ; ALWAYS jump to valid path (EB 00)
  0040120C: mov   eax, 1                 ; always reaches here now
  00401211: pop   ebp
  00401212: ret

; Patch bytes at 0x0040120A:
; Before: 74 24  (JZ rel8 +0x24)
; After:  EB 00  (JMP rel8 +0x00 = next instruction)

Applying the patch

bash 
# Method 1: hex editor
xxd sample.exe | grep -n '74 24'   # find the bytes
# Open in hexedit, navigate to offset, change 74 24 → EB 00

# Method 2: Python
python3 - << 'EOF'
with open('sample.exe', 'r+b') as f:
    f.seek(0x40120A)        # file offset (not virtual address)
    original = f.read(2)
    print(f'Original bytes: {original.hex()}')
    f.seek(0x40120A)
    f.write(b'\xEB\x00')    # JMP +0 (skip to next instruction)
    print('Patch applied')
EOF

# Method 3: Ghidra patch
# Right-click instruction → Patch Instruction
# Or use the Patch Bytes option and enter new bytes directly
# File → Export Program → Original File (with patched bytes)

# Method 4: x64dbg (dynamic patch)
# Navigate to address, right-click → Assemble
# Type: jmp 0x0040120C
# File → Patch File → Save patched executable

# Verify patch:
cmp original_sample.exe patched_sample.exe
diff <(xxd original_sample.exe) <(xxd patched_sample.exe)

06 Packed and Obfuscated Samples

Most real-world malware is packed — the original code is compressed or encrypted and unpacked at runtime. The outer binary is just a loader. Indicators of packing:

bash 
# Identify packing
file suspicious.exe
# Packed EXEs often show: UPX compressed, or 'MS-DOS executable' with suspicious entropy

# Check entropy (high entropy = likely packed/encrypted)
die suspicious.exe    # 'Detect It Easy' — identifies packer by signature
rabin2 -S suspicious.exe | awk '{print $5,$6}' | sort -k2 -n  # section entropy
# Entropy >7.0 in .text section = suspicious; normal code is ~6.0-6.5

# String output barely readable = packed
strings -n 6 suspicious.exe | wc -l   # very low count suggests packing

# UPX unpack (if packer is UPX — most common)
upx -d suspicious.exe -o unpacked.exe
file unpacked.exe    # should now show PE32 executable normally

# If custom packer: use dynamic approach
# 1. Run in debugger
# 2. Set breakpoint on VirtualAlloc / VirtualProtect
# 3. The packer allocates memory, decrypts code into it, then jumps to it
# 4. Break when execution jumps to the allocated region
# 5. Dump the unpacked code from memory

# x64dbg: Scylla plugin — dump unpacked PE from memory
# Plugins → Scylla → Dump (after reaching OEP)

07 Malware Analysis Walkthrough — Extracting a C2 Address

A condensed real-world workflow: unknown sample, need to extract the C2 address without running the malware on an internet-connected machine.

Step 1 — Triage

bash 
# Initial triage — safe, no execution
file sample.bin
# → PE32+ executable (GUI) x86-64, MS Windows

md5sum sample.bin; sha256sum sample.bin
# Hash for threat intel lookup

# Check VirusTotal hash (if not client-confidential)
curl -s 'https://www.virustotal.com/api/v3/files/SHA256HASH' \
  -H 'x-apikey: YOUR_KEY' | python3 -m json.tool | grep -A3 'last_analysis_stats'

# Strings — look for URLs, IPs, registry paths, mutex names
strings -n 8 sample.bin | grep -iE 'http|cmd|powershell|user-agent|\\SOFTWARE'
# → User-Agent: Mozilla/5.0 (compatible; MSIE 9.0)
# → http://[REDACTED]/update
# → SOFTWARE\Microsoft\Windows\CurrentVersion\Run

# Imports — what Win32 APIs does it call?
dump-pe.py sample.bin --imports | grep -iE 'virtualalloc|createprocess|socket|connect|winhttp|urldownload'
# Heavy network imports + VirtualAlloc + CreateRemoteThread = classic dropper/RAT

Step 2 — Static analysis in Ghidra

c 
/* Ghidra decompiler — network initialisation function (after renaming) */
void init_c2_connection(void) {
    HINTERNET hSession;
    HINTERNET hConnect;
    char c2_host [256];
    char decrypted [256];
    
    /* XOR decode the C2 hostname from embedded blob */
    xor_decode(&DAT_0040a100, 0x12, decrypted, 0x12);  /* key = 0x12 */
    
    hSession = WinHttpOpen(
        L"Mozilla/5.0 (compatible; MSIE 9.0)",  /* custom UA */
        WINHTTP_ACCESS_TYPE_DEFAULT_PROXY,
        NULL, NULL, 0
    );
    
    hConnect = WinHttpConnect(
        hSession,
        (LPCWSTR)decrypted,  /* XOR-decoded hostname */
        443,
        0
    );
    
    /* Beacon loop follows */
    beacon_loop(hConnect);
}

Step 3 — Decrypt the embedded C2 hostname

python 
#!/usr/bin/env python3
# Extract XOR-encoded C2 hostname from sample
# Identified in Ghidra: DAT_0040a100, length 0x12, key 0x12

with open('sample.bin', 'rb') as f:
    data = f.read()

# Convert virtual address to file offset
# For a .data section at VA 0x0040a000, file offset 0x8200:
VA_OFFSET  = 0x0040a000
FILE_OFFSET = 0x8200

data_va = 0x0040a100
file_offset = FILE_OFFSET + (data_va - VA_OFFSET)

encoded = data[file_offset : file_offset + 0x12]
xor_key = 0x12

decoded = bytes([b ^ xor_key for b in encoded])
print(f'Encoded: {encoded.hex()}')
print(f'Decoded: {decoded}')
# → Decoded: b'c2.evil-domain.com'

# Verify by checking the decoded string is a valid hostname
import re
if re.match(r'^[a-zA-Z0-9.-]+\.[a-zA-Z]{2,}$', decoded.decode('ascii', errors='ignore')):
    print('[+] Valid hostname extracted — confirmed C2')

08 Useful x64dbg Techniques

text 
Essential x64dbg workflow for dynamic analysis:

── BREAKPOINTS ───────────────────────────────────────────────────
F2              Set/toggle breakpoint at current instruction
Ctrl+G          Go to address or symbol (type: VirtualAlloc)
bp VirtualAlloc  Break when VirtualAlloc is called (API breakpoint)
bp WS2_32.connect  Break on network connect

── EXECUTION ─────────────────────────────────────────────────────
F9              Run (continue execution)
F8              Step over (don't enter function calls)
F7              Step into
F4              Run to cursor
Ctrl+F9         Run until return (execute rest of current function)

── INSPECTION ────────────────────────────────────────────────────
Ctrl+D          View in dump (memory at selected address)
Ctrl+R          Find references to selected address
Alt+M           Memory map — see all allocated regions
Alt+B           Breakpoint list
Ctrl+L          Log window

── ANTI-ANALYSIS BYPASS ──────────────────────────────────────────
  Most malware checks:
  - IsDebuggerPresent → patch to return 0 or set EAX=0 after call
  - CheckRemoteDebuggerPresent → same approach
  - Timing checks (RDTSC) → NOP the conditional jump after
  - Plugin: ScyllaHide — automatically patches common anti-debug techniques

── DUMP MEMORY ───────────────────────────────────────────────────
  After unpacking:
  1. Find the OEP (Original Entry Point) — break when execution leaves
     the packer stub and enters the real code
  2. Plugins → Scylla → Dump (choose OEP address)
  3. Fix imports → IAT Autosearch → Get Imports → Fix Dump
  4. Save: fully functional unpacked executable
