Q1 2026 Threat Landscape – Zero-Days, MDM Campaigns & AD Privilege Paths

2026-06-03 · 4 min read · News · cve zero-day active-exploitation ivanti active-directory linux

Q1 2026 brought a wave of pre-authentication zero-days exploited before public disclosure, paired MDM campaigns against Ivanti EPMM, a critical Citrix NetScaler auth bypass, a Netlogon privilege path to domain control, and a public Linux kernel LPE. None of these overlap with the Q2 post.

01 What Set Q1 Apart

Q1 2026 introduced a pattern that would define the rest of the year: vulnerabilities exploited before anyone outside the vendor knew they existed. Several CVEs entered the CISA KEV catalogue within days of first public disclosure — meaning defenders had effectively zero reaction time.

The dominant campaigns were not opportunistic. Threat actors ran coordinated, multi-CVE operations against the same platform (Ivanti EPMM) using two separate zero-days in tandem, deploying web shells and backdoors before patches were available.

02 CVE-2026-1281 — Ivanti EPMM Pre-Auth RCE (Critical, Zero-Day)

Type: Unauthenticated remote code execution — code injection
CVSS: 9.8 | Exploitation: Confirmed active, in KEV, exploited before disclosure
Attack chain role: Initial access → system compromise → persistence

This is the first of two paired zero-days hitting Ivanti Endpoint Manager Mobile in Q1. The root cause is a code injection flaw in the In-House Application Distribution and Android File Transfer configuration features — both reachable without any credentials from the internet.

What makes this particularly severe is the pre-disclosure exploitation window: threat actors had a working exploit before the vendor had issued any advisory, before any patch existed, and before any defender had reason to check. Mass internet scanning for exposed EPMM instances was observed within the same campaign window.

Why it matters: EPMM is the administrative control plane for every enrolled mobile device — phones, tablets, laptops. Pre-auth RCE on this platform is equivalent to handing an attacker an enterprise-wide device management console on day zero.

bash 
# Identify exposed Ivanti EPMM instances (check your own assets)
nmap -p 443,8443 --script http-title,ssl-cert <target-range> \
  | grep -iE 'mobileiron|ivanti|mifs'

# Version fingerprint (unauthenticated endpoint on unpatched builds)
curl -sk 'https://<target>/mifs/aad/rp/content/images/devices.json' -I
curl -sk 'https://<target>/api/v1/metadata/version' | python3 -m json.tool

# Detection: hunt for unexpected processes spawned by EPMM service
# On the server -- look for shells launched by the Java process
ps auxf | grep -A5 'java.*epmm\|java.*mifs'

# Hunt web shells dropped post-exploitation
find /opt/MobileIron /opt/mifs /usr/local/mobilemanager \
  -name '*.jsp' -newer /etc/passwd -ls 2>/dev/null

# Check outbound connections from EPMM host (C2 indicator)
ss -tnp | grep -v '127.0.0.1'
netstat -anp | grep ESTABLISHED | grep java

03 CVE-2026-1340 — Ivanti EPMM Pre-Auth RCE (Critical, Zero-Day)

Type: Unauthenticated remote code execution — code injection
CVSS: 9.8 | Exploitation: Confirmed active, in KEV, public PoC available
Attack chain role: Initial access (operationally paired with CVE-2026-1281)

This CVE was observed within the same exploitation campaigns as CVE-2026-1281 — not as an alternative but as a complement. Attackers used both vulnerabilities against the same targets in the same intrusion windows, likely as reliability redundancy: if one payload path failed, the other succeeded.

The pairing matters for detection: blocking one CVE's specific request pattern is not sufficient. Defenders who patched one but not both remained exposed.

bash 
# Both CVEs share the same detection surface -- check EPMM API access logs
# Look for unauthenticated requests to sensitive API paths:
grep -E '(POST|GET).*/mifs/aad|/api/v1/(apps|devices|users|config)' \
  /var/log/nginx/access.log 2>/dev/null \
  | grep -v '200\|302' | tail -50

# Check for newly created admin accounts (post-exploitation persistence)
curl -sk 'https://localhost/api/v2/administrators' \
  -H 'Authorization: Bearer <admin-token>' | python3 -m json.tool \
  | grep -E 'created_at|username|role'

# Hunt for backdoor installations -- unusual binaries in EPMM dirs
find /opt /usr/local -newer /etc/passwd -type f -executable \
  ! -path '*/proc/*' 2>/dev/null | grep -vE '\.(jar|class|war)$'

# Verify patch level against vendor advisory
# Patched versions: EPMM >= 11.12.0.4 / 11.11.0.6 / 11.10.0.6
curl -sk 'https://<target>/api/v1/metadata/version' \
  | python3 -c "import json,sys; d=json.load(sys.stdin); print('Version:', d.get('productVersion','unknown'))"
The MDM Campaign Pattern — Three CVEs, One Platform:

Q1 2026 established a sustained campaign against Ivanti EPMM using CVE-2026-1281 and CVE-2026-1340 in tandem for initial access, followed later in Q2 by CVE-2026-6973 (authenticated RCE) to deepen access after credentials were stolen.

This is a full attack chain built around one product: get in pre-auth → steal admin creds → exploit authenticated path for persistence and deeper control. If your organisation runs any version of Ivanti EPMM / MobileIron, treat all three CVEs as a coordinated campaign, not isolated bugs.

04 CVE-2026-3055 — Citrix NetScaler ADC / Gateway (Critical)

Type: Authentication bypass / session abuse on the NetScaler management and VPN interface
CVSS: 9.3 | Exploitation: Confirmed active, in KEV
Attack chain role: Initial access → authentication abuse → network entry

NetScaler ADC and Gateway are the perimeter devices organisations trust for VPN access, SSO, and SAML-based authentication for remote workforces. This vulnerability allows an attacker to abuse the authentication flow — bypassing it or hijacking sessions — without needing valid credentials.

Why it matters: A NetScaler compromise happens before any internal control fires. Downstream SIEM rules that alert on "suspicious internal login" assume the perimeter authenticated the user correctly. When the perimeter itself is the vulnerability, that assumption breaks.

Observed activity included internet-wide scanning specifically targeting unpatched NetScaler appliances, followed by perimeter reconnaissance and authentication targeting — consistent with nation-state and ransomware precursor activity.

bash 
# Fingerprint exposed NetScaler instances
curl -sk 'https://<target>/vpn/index.html' | grep -iE 'citrix|netscaler|ns_build'
curl -sk 'https://<target>/nitro/v1/config/nsversion' \
  -H 'X-NITRO-USER: nsroot' -H 'X-NITRO-PASS: nsroot' 2>/dev/null

# Shodan query (authorised asset discovery)
# shodan search 'http.title:"Citrix Gateway" ssl.cert.issuer.cn:Citrix'

# Check NS build version (via management interface, authenticated)
ssh nsroot@<netscaler> 'show version' 2>/dev/null | grep 'NetScaler'

# Detection: look for abnormal session patterns in ns.log
# Typically at /var/log/ns.log or /var/nslog/
grep -iE '(SSLVPN|AAA|SAML).*(ERROR|FAIL|bypass|invalid)' /var/log/ns.log | tail -30

# Hunt for unexpected admin sessions from unusual IPs
grep 'Login succeeded' /var/log/ns.log \
  | awk '{print $NF}' | sort | uniq -c | sort -rn | head -20

# Check if SAML metadata has been tampered with
curl -sk 'https://<target>/saml/login' -I | grep -i 'saml\|idp\|sp'

05 CVE-2026-41089 — Windows Server Netlogon Privilege Escalation (Critical)

Type: Privilege escalation in the Windows Netlogon Remote Protocol
CVSS: 9.8 | Exploitation: Observed in the wild
Attack chain role: Privilege escalation → SYSTEM access → domain control → Kerberos abuse

Netlogon is the Windows protocol domain-joined machines use to authenticate against domain controllers. A flaw here means an attacker with any foothold inside the network — even a low-privilege user account — can escalate toward SYSTEM on a domain controller.

From SYSTEM on a DC, the game is over: the attacker can dump the NTDS.dit (all domain password hashes), forge Kerberos tickets (Golden Ticket), create persistent backdoor domain admin accounts, and own every domain-joined machine.

Attack path in observed intrusions:
Low-priv domain user → CVE-2026-41089 → SYSTEM on DC → DCSync → full domain compromise

powershell 
# --- Detection on Domain Controllers ---

# Check Netlogon service status and version
Get-Service -Name Netlogon | Select-Object Status, StartType
(Get-Item 'C:\Windows\System32\netlogon.dll').VersionInfo.FileVersion

# Hunt for exploitation artefacts: unexpected SYSTEM-level processes
Get-Process | Where-Object {$_.SI -eq 0} | Select-Object Name, Id, Path

# Check for new privileged accounts created recently (post-exploit persistence)
Get-ADUser -Filter * -Properties WhenCreated, MemberOf |
  Where-Object {$_.WhenCreated -gt (Get-Date).AddDays(-14)} |
  Select-Object SamAccountName, WhenCreated, Enabled

# Check Domain Admins group for unexpected members
Get-ADGroupMember -Identity 'Domain Admins' |
  Select-Object SamAccountName, distinguishedName

# Hunt for DCSync activity in Security event log
# Event 4662: Operation performed on an object (look for replication rights)
Get-WinEvent -FilterHashtable @{LogName='Security'; Id=4662} |
  Where-Object {$_.Message -match '1131f968|1131f6ad|89e95b76'} |
  Select-Object TimeCreated, Message | Select-Object -First 10

# Check Netlogon log for anomalous entries
Get-Content 'C:\Windows\debug\netlogon.log' | Select-Object -Last 100 |
  Where-Object {$_ -match 'ERROR\|FAILED\|DENIED'}
bash 
# --- Red team / lab perspective (authorised AD testing) ---

# Check if DC is vulnerable via version (requires domain access)
crackmapexec smb <dc-ip> -u user -p pass --gen-relay-list /tmp/relay.txt

# Enumerate DC via Netlogon (requires domain creds)
python3 -c "
from impacket.dcerpc.v5 import nrpc, transport
# Connect to Netlogon on DC
"

# Post-compromise: verify domain admin access achieved
crackmapexec smb <dc-ip> -u Administrator -H <ntlm-hash> --shares
secretsdump.py -just-dc DOMAIN/Administrator@<dc-ip> -hashes :<ntlm-hash>

# DCSync to dump all hashes (confirms domain compromise)
secretsdump.py -just-dc-ntlm DOMAIN/Administrator@<dc-ip> 2>/dev/null | head -20

06 CVE-2026-31431 — Linux Kernel "Copy Fail" Local Privilege Escalation

Type: Local privilege escalation in the Linux kernel
Exploitation: Public exploit available
Attack chain role: Post-exploitation — converts limited user access to root
Alias: "Copy Fail"

This kernel vulnerability allows a local user (including a container user with certain capabilities, or a process with limited shell access) to escalate to root. It does not provide initial access on its own — but paired with any low-privilege foothold (SSH with a weak credential, a web shell running as www-data, a misconfigured sudo entry), it becomes a clean root escalation.

Affected distributions include Ubuntu, RHEL, SUSE, and Amazon Linux — meaning this affects both on-premise Linux servers and cloud workloads.

Why it matters to attackers: Web shells and initial access often land in unprivileged contexts. "Copy Fail" turns that limited foothold into full root without any network movement or credential theft.

bash 
# Check kernel version (vulnerability window varies by distro/patch)
uname -r
cat /etc/os-release

# Ubuntu patched kernel versions (example)
# Ubuntu 22.04: 5.15.0-105+
# Ubuntu 20.04: 5.4.0-182+
# RHEL 9: kernel-5.14.0-427+

# Check if running a vulnerable kernel (generic version check)
kernelver=$(uname -r)
echo "Running kernel: $kernelver"
# Compare against vendor advisory for your distro

# Check if public PoC applies (test in lab only)
# PoC typically exploits copy_file_range() syscall mishandling
# Indicator: any setuid binary writable or exploitable via the copy path
find / -perm -4000 -type f 2>/dev/null | head -20

# Detection: kernel audit log for unusual privilege escalation
ausearch -m avc,user_avc -ts recent 2>/dev/null | grep -i 'execve\|capability'

# Check for exploitation artefacts: new SUID binaries
# Baseline: run this before and after any incident
find / -perm -4000 -type f 2>/dev/null | sort > /tmp/suid_now.txt
diff /tmp/suid_baseline.txt /tmp/suid_now.txt

# Monitor for unexpected root processes from web service users
auditctl -a always,exit -F arch=b64 -S execve -F uid=33 -k www-exec  # uid 33 = www-data
audit2allow -la | grep www-exec

07 Campaign Patterns Across Q1

Three distinct campaign patterns emerged from Q1 activity:

Pattern 1: MDM Platform Targeting
CVE-2026-1281 and CVE-2026-1340 were not independent incidents — they were used together in the same operations. Attackers who compromised EPMM via CVE-2026-1281 often had CVE-2026-1340 payloads staged as a fallback. The goal was reliable initial access to mobile management infrastructure, followed by web shell persistence and credential harvesting for lateral movement.

Pattern 2: Identity & Access Infrastructure
CVE-2026-3055 (Citrix NetScaler) and CVE-2026-0257 (Palo Alto GlobalProtect, Q2) both target the same strategic layer — the network perimeter where authentication decisions are made. Compromising this layer means attackers arrive in the internal network already authenticated, making subsequent activity invisible to tools that trust perimeter decisions.

Pattern 3: Post-Compromise Privilege Escalation
CVE-2026-41089 (Netlogon) and CVE-2026-31431 (Linux kernel) serve a different role — they are not entry points but multipliers. Once any low-privilege foothold exists, these convert it to domain admin or root. Their presence in active campaigns signals that attackers had already achieved initial access and were escalating.

Q1 Key Takeaways:
  • Pre-disclosure zero-days are not rare anymore — CVE-2026-1281 was being exploited while the vendor was still unaware. Patch SLAs measured in days, not weeks.
  • Paired CVEs mean patching one is not enough — CVE-2026-1281 and CVE-2026-1340 were designed to be used together. Always check for sibling CVEs on the same product.
  • Web shell + backdoor = the attacker is staying — observed post-exploitation on EPMM was persistence-first, not data-first. Assume dwell time; hunt retroactively.
  • CVE-2026-31431 needs no network access — any web shell, any low-priv SSH session, any misconfigured SUID can trigger it. Patch Linux kernels across all workloads including cloud.
