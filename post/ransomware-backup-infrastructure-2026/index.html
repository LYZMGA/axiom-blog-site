Ransomware Affiliates Are Hunting Your Backups First — The Veeam RCE Chain

2026-06-17 · 2 min read · News · ransomware veeam backup cve active-exploitation active-directory

Two critical Veeam Backup & Replication CVEs (CVSS 9.4 and 9.9) let an attacker pivot from a single domain account to wiping the restore path entirely. Why backup infrastructure has become the first target in the modern ransomware kill chain, not the last.

Ransomware-as-a-Service affiliates have learned a lesson the hard way for defenders: encryption alone is reversible if the victim has working backups. So the modern playbook inverts the order of operations — neutralise the recovery path before detonating the payload. Backup infrastructure is no longer the safety net behind the perimeter. It is frequently the first internal target.

March 2026 made this concrete. Veeam disclosed seven critical vulnerabilities in Backup & Replication, the two most severe of which — CVE-2026-44963 and CVE-2026-21708 — let an attacker go from an ordinary domain account to code execution on the backup server itself. CISA added Veeam flaws to the Known Exploited Vulnerabilities catalogue again, continuing a multi-year pattern: ransomware crews specifically chase Veeam disclosures. Veeam is simply the most visible target — the same affiliates run identical reconnaissance against Acronis Cyber Protect and Windows Server Backup, but Veeam's market share keeps it at the top of the list.

The Two CVEs That Matter

CVECVSSMechanismAffected / Fixed
CVE-2026-449639.4RCE on the Backup Server, triggerable by any **authenticated domain user** — no local admin or special group membership required12.3.2.4465 and earlier 12.x builds. **Not exploitable on 13.x** — architectural changes in v13 close this path. Fixed in 12.3.2.4854
CVE-2026-217089.9Backup Viewer component allows RCE running as the `postgres` service account — effectively root over the configuration database12.3.2.4165 and earlier. Also affected early **13.0.x builds**. Fixed in 12.3.2.4465 and 13.0.1.2067
A correction worth dwelling on

CVE-2026-44963 needs an authenticated domain account — not an anonymous or unauthenticated actor. That distinction matters operationally: it means any phished low-privilege user, any compromised service account, any foothold at all on the domain is sufficient to reach the backup server. You do not need privilege escalation first. The backup server is the privilege escalation.

Why This Keeps Happening — The Convergence Anti-Pattern

The root cause is architectural, not a single bad patch. In the overwhelming majority of audited environments, the backup management server is joined directly to the production Active Directory domain it protects. The moment an attacker holds Domain Admin — or, as CVE-2026-44963 demonstrates, sometimes nothing more than a standard domain account — they inherit a path straight into the one system whose entire purpose is surviving the attacker's other objective.

Enterprise backup platforms also lean on relational database backends (PostgreSQL for Veeam, frequently MS SQL elsewhere) to store repository credentials and retention policy. Compromise that database and an attacker doesn't need to touch the actual backup files — they can simply rewrite the retention policy or truncate the catalogue so the data silently expires.

Exploitation Mechanics

Veeam's Backup service listens by default on TCP/9392. The RCE chain in CVE-2026-44963 involves an authenticated client sending a crafted request that triggers a deserialization flaw in the service, spawning a process under NT AUTHORITY\SYSTEM. From there, an attacker reaches for the database utilities already installed alongside the product — psql.exe for the PostgreSQL backend that ships with most Veeam installs — to manipulate retention policy or purge backup catalogue entries directly, bypassing the application's own retention safeguards.

bash 
# Identify exposed/internal Veeam infrastructure (authorised asset discovery)
nmap -p 9392,9401,9419,9420 --open <your-ranges>

# Confirm version against the fixed builds
# 12.x must be >= 12.3.2.4854 to be clear of CVE-2026-44963
# 12.x must be >= 12.3.2.4465 to be clear of CVE-2026-21708
# 13.x must be >= 13.0.1.2067 to be clear of CVE-2026-21708

# Check CISA KEV for current Veeam entries
curl -s https://www.cisa.gov/sites/default/files/feeds/known_exploited_vulnerabilities.json \
  | python3 -c "import json,sys; [print(e['cveID'], e['shortDescription']) for e in json.load(sys.stdin)['vulnerabilities'] if 'veeam' in e.get('vendorProject','').lower()]"

Architectural Hardening Blueprint

1. Identity isolation. Backup management servers should live in an isolated administrative workgroup, or — better — a dedicated single-purpose management AD forest with no trust relationship to the production domain. A domain user account anywhere in production should never be a credential that means anything to the backup server.

2. True immutability. Hardened Repositories on Linux with single-use credentials (so even a compromised backup server cannot delete already-written blocks), or Object Lock in compliance mode on S3/Azure Blob with a retention period that cannot be shortened by any administrative account, including your own.

3. Patch the backup software like it's a domain controller, not like it's a server that backs up domain controllers. The SLA should be the same.

Detection Engineering

yaml 
title: Database Utility Execution Against Veeam Backend by Unexpected Parent
id: 5b6c8b41-e9de-4b11-be83-9b62a67bc3ef
status: stable
description: |
    Detects execution of psql.exe or sqlcmd.exe referencing the Veeam database
    when the parent process is not the legitimate Veeam backup service itself.
    Following CVE-2026-44963 / CVE-2026-21708, attackers who gain code execution
    on the backup server commonly pivot to the bundled PostgreSQL utilities to
    manipulate retention policy or purge backup history.
references:
    - https://thehackernews.com/2026/03/veeam-patches-7-critical-backup.html
    - https://www.veeam.com/kb4696
tags:
    - attack.impact
    - attack.t1490
logsource:
    category: process_creation
    product: windows
detection:
    selection_utilities:
        Image|endswith:
            - '\psql.exe'
            - '\sqlcmd.exe'
    selection_arguments:
        CommandLine|contains:
            - 'veeam'
            - 'VeeamBackup'
    filter_legitimate:
        ParentImage|endswith: '\Veeam.Backup.Service.exe'
    condition: (selection_utilities and selection_arguments) and not filter_legitimate
falsepositives:
    - Database maintenance scripts run by designated DBAs through documented change windows
level: critical
Red team value

Demonstrating the pivot from a single low-privilege domain account to a shell on the backup server is one of the most consequential findings you can put in a report — it directly quantifies the gap between "we have backups" and "we have backups an attacker with domain access cannot also destroy." If your test environment has a Veeam instance joined to the assessed domain, this should be on the path list by default.
