OSINT — Open Source Intelligence from First Principles

2026-06-04 · 4 min read · Security · osint recon google-dorks shodan red-team reconnaissance security

OSINT methodology, Google dorks, Shodan, email and username enumeration, social media investigation, domain and IP recon, image geolocation, document metadata, GitHub secrets, dark web sources, and building target profiles for red team engagements.

OSINT — Open Source Intelligence — is the systematic collection and analysis of information from publicly available sources. It is the foundation of every engagement: reconnaissance determines your attack surface before a single packet reaches the target. Done well, OSINT tells you the employee's name, email format, password reuses, exposed admin panels, internal infrastructure details, and the pretext you need to walk in the front door.

This guide covers every major technique and tool — from Google dorks to Shodan, email enumeration to image geolocation — with the red team context that turns raw data into actionable intelligence.

01 The OSINT Mindset and Methodology

OSINT is not random Googling. It is a structured process:

text 
1. Define objective     — what question am I answering?
2. Identify sources     — where is this data likely to exist?
3. Collect passively    — no direct contact with target systems
4. Analyse and pivot    — each piece of data opens new queries
5. Document findings    — everything timestamped and sourced
6. Assess confidence    — single-source vs corroborated

Passive vs active recon:

TypeDescriptionDetectability
**Passive**Public sources only — search engines, WHOIS, social media, leaked databasesZero — target never sees you
**Semi-passive**Normal-looking traffic to target — browsing the website, reading public APIVery low
**Active**Direct probing — port scanning, DNS zone transfers, directory brute forceDetectable

OSINT is passive and semi-passive. The moment you send a packet to the target's infrastructure you cross into active reconnaissance.

The pivot loop:

Every finding enables new searches. An email address reveals a name. A name reveals a LinkedIn. LinkedIn reveals an employer history. An employer reveals a domain. A domain reveals an IP range. An IP range reveals exposed services.

text 
email → name → social profiles → employer → domain
   ↓                                            ↓
password leaks                           Shodan / Censys
   ↓                                            ↓
breached services                       exposed admin panels

OSINT frameworks:

  • OSINT Framework — osintframework.com — interactive directory of every OSINT tool by category
  • MITRE ATT&CK Reconnaissance — TA0043 — formal taxonomy of reconnaissance techniques
  • PTES (Penetration Testing Execution Standard) — intelligence gathering phase

02 Google Dorking

Google's advanced search operators let you construct precise queries that surface information the site owner didn't intend to expose. This is called Google Dorking or Google Hacking.

Core operators

OperatorFunctionExample
`site:`Restrict to domain`site:target.com`
`filetype:`Specific file extension`filetype:pdf site:target.com`
`inurl:`String in URL`inurl:admin site:target.com`
`intitle:`String in page title`intitle:"index of" site:target.com`
`intext:`String in body text`intext:"confidential" site:target.com`
`cache:`Google's cached copy`cache:target.com`
`link:`Pages linking to URL`link:target.com`
`related:`Similar sites`related:target.com`
`""`Exact phrase`"john smith" "target corp"`
`-`Exclude term`site:target.com -www`
`*`Wildcard`"target * password"`
`OR` / ``Either term`site:target.com OR site:target.net`
`..`Number range`"2020..2025" site:target.com`

High-value dorks for red team recon

text 
# Exposed admin panels
site:target.com inurl:admin
site:target.com inurl:login
site:target.com intitle:"admin panel"
site:target.com inurl:wp-admin
site:target.com inurl:phpmyadmin

# Config and credential files
site:target.com filetype:env
site:target.com filetype:cfg
site:target.com filetype:conf
site:target.com filetype:xml inurl:config
site:target.com ext:sql
site:target.com ext:bak
site:target.com ext:log

# Directory listings
site:target.com intitle:"index of"
site:target.com intitle:"index of" "parent directory"
site:target.com intitle:"index of" passwd

# Exposed documents with sensitive content
site:target.com filetype:pdf "confidential"
site:target.com filetype:xlsx
site:target.com filetype:docx

# Subdomains discovery
site:*.target.com
site:*.target.com -site:www.target.com

# Error messages disclosing info
site:target.com "SQL syntax"
site:target.com "Warning: mysql_"
site:target.com "Fatal error"

# VPN / remote access
site:target.com inurl:vpn
site:target.com inurl:remote
site:target.com intitle:"Citrix"
site:target.com intitle:"SSL VPN"

# Email addresses
site:target.com "@target.com"
site:target.com intext:"@target.com" filetype:pdf

# Passwords in plaintext
site:target.com intext:password filetype:log
site:target.com intext:"password" filetype:txt

03 Shodan — The Search Engine for Devices

Shodan continuously scans the internet and indexes the banners returned by every open port on every IP address. It is the single most powerful passive reconnaissance tool against infrastructure.

Account and API setup

bash 
# Install CLI
pip3 install shodan

# Initialise with your API key (free account: 100 queries/month)
shodan init YOUR_API_KEY

# Test
shodan info

Web interface queries (shodan.io)

text 
# Find everything for an organisation
org:"Target Corporation"

# Find by domain
hostname:target.com

# Find by IP range (your own / authorised)
net:10.10.10.0/24

# Find specific products
product:"Apache httpd"
product:"nginx"
product:"Microsoft IIS"

# Find by port
port:3389 org:"Target"
port:22 org:"Target"
port:5900 org:"Target"   # VNC

# Find exposed industrial control systems
port:102 product:"Siemens"   # S7
country:US port:44818         # EtherNet/IP (Rockwell)
port:2404                     # IEC 60870

# Find specific services
product:"Elasticsearch" port:9200
product:"MongoDB" port:27017
product:"Redis" port:6379

# Find login pages / admin panels
http.title:"Admin"
http.title:"Login"
http.title:"phpMyAdmin"

# Find by SSL certificate
ssl.cert.subject.CN:"*.target.com"
ssl:"target.com"

# Find by specific header or content
http.html:"target.com"
http.favicon.hash:-1555085002   # Favicon hash (find same software)

# Find default credentials
http.title:"router" default password

CLI usage

bash 
# Search and get summary
shodan search 'org:"Target" port:443'

# Get detailed host info
shodan host 192.168.1.1

# Download all results
shodan download results 'org:"Target"'
shodan parse --fields ip_str,port,transport results.json.gz

# Count results
shodan count 'org:"Target"'

# Find IPs for a domain
shodan domain target.com

# Monitor your own assets
shodan alert create mynetwork 192.168.1.0/24

# myip
shodan myip

04 Domain and DNS Reconnaissance

A domain name is the starting point for infrastructure mapping. Extract every piece of data associated with it before touching active tools.

WHOIS

bash 
# Basic WHOIS
whois target.com
whois 192.168.1.1

# Registrar, registration date, expiry, registrant contact (if not privacy-protected)
# Key fields: Registrant Name, Email, Organisation, Name Servers

# Online WHOIS with history
# https://lookup.icann.org
# https://www.whois.com
# https://whoishistory.com   (historical — pivotal for finding old emails)
# https://domaintools.com    (paid, best for pivot)

DNS enumeration

bash 
# Basic DNS lookup
nslookup target.com
nslookup -type=MX target.com
nslookup -type=NS target.com
nslookup -type=TXT target.com  # SPF, DKIM, verification tokens — leaks service providers
nslookup -type=ANY target.com

# dig
dig target.com A
dig target.com MX
dig target.com NS
dig target.com TXT
dig target.com AAAA     # IPv6
dig target.com SOA      # Start of Authority — primary nameserver + admin email
dig target.com CNAME

# Reverse DNS
dig -x 192.168.1.1

# DNS zone transfer (if misconfigured — gold mine)
dig @ns1.target.com target.com AXFR
nslookup -type=AXFR target.com ns1.target.com

# Zone transfer exposes ALL DNS records — subdomains, internal hosts, mail servers
# Rare on modern infrastructure but common in older organisations

SPF/DKIM/DMARC — intelligence from mail records

bash 
# SPF record — lists every mail service used
dig target.com TXT | grep spf
# Example: v=spf1 include:_spf.google.com include:sendgrid.net include:mailchimp.com ~all
# Reveals: Google Workspace, SendGrid for marketing, Mailchimp for newsletters

# DMARC
dig _dmarc.target.com TXT
# p=none means no enforcement — phishing from @target.com is harder to auto-reject

# DKIM selector discovery
dig google._domainkey.target.com TXT
dig mail._domainkey.target.com TXT

Subdomain enumeration

bash 
# Passive — no DNS queries to target
# crt.sh (Certificate Transparency logs)
curl -s "https://crt.sh/?q=%.target.com&output=json" | jq -r '.[].name_value' | sort -u

# subfinder (passive, aggregates multiple sources)
subfinder -d target.com
subfinder -d target.com -all -o subdomains.txt

# amass (passive + active, most comprehensive)
amass enum -passive -d target.com
amass enum -d target.com -o subdomains.txt

# theHarvester (passive, also gets emails)
theHarvester -d target.com -b all
theHarvester -d target.com -b google,bing,shodan,certspotter

# Active DNS brute force (crosses into active recon)
gobuster dns -d target.com -w /opt/SecLists/Discovery/DNS/subdomains-top1million-5000.txt
ffuf -u http://FUZZ.target.com -w /opt/SecLists/Discovery/DNS/subdomains-top1million-5000.txt -mc 200,301,302

# dnsrecon
dnsrecon -d target.com -t std      # standard records
dnsrecon -d target.com -t brt -D subdomains.txt  # brute force
dnsrecon -d target.com -t axfr     # zone transfer attempt

Certificate Transparency (crt.sh)

Every SSL certificate issued by a public CA is logged in Certificate Transparency logs. This reveals subdomains the organisation never intended to make public:

bash 
# Web
https://crt.sh/?q=%.target.com

# API
curl -s 'https://crt.sh/?q=%.target.com&output=json' | \
  jq -r '.[].name_value' | \
  sed 's/\*\.//g' | \
  sort -u | \
  grep -v '^\.' > crt_subdomains.txt

# What you find: dev.target.com, staging.target.com, vpn.target.com,
#               jira.target.com, jenkins.target.com, old.target.com

ASN and IP range enumeration

bash 
# Find ASN for organisation
whois -h whois.radb.net target.com
bgp.he.net/search?search[search]=Target+Corp

# Get all IP ranges for an ASN
whois -h whois.radb.net -- '-i origin AS12345' | grep -E '^route:'

# Or use amass
amass intel -org "Target Corporation"
amass intel -asn 12345

Deriving usernames from names

Given a target's full name, generate all common username formats:

bash 
# namemash.py
python3 namemash.py 'John Smith' > john_usernames.txt
# Outputs: jsmith, smithj, john.smith, johnsmith, smith.john, j.smith...

# Manual patterns
name="john" surname="smith"
for u in "$name$surname" "$surname$name" "${name:0:1}$surname" "$name.${surname:0:1}" \
  "${name:0:1}.$surname" "$name.$surname" "$surname.$name" \
  "$name\_$surname" "$name${surname:0:1}"; do
  echo "$u"
done

09 Document and Metadata OSINT

Documents published by an organisation often contain hidden metadata revealing internal structure, software versions, and employee names.

FOCA — Fingerprinting Organisations with Collected Archives

FOCA (Windows tool) automatically downloads and extracts metadata from documents found on a domain:

text 
- Authors, editors, last-modified-by
- Software versions (Office 2013, Adobe Acrobat 9)
- Internal hostnames (saved path: \\FILESERVER01\docs\report.docx)
- Printer names
- Internal email addresses

Manual metadata extraction

bash

17 Legal and Ethical Boundaries

OSINT uses only public data — but that doesn't mean all OSINT is legal or ethical in all contexts.

What is clearly legal:
- Searching public websites, social media, company filings
- Using public databases (WHOIS, crt.sh, Shodan)
- Reading publicly available breach data for defensive purposes
- Conducting authorised red team reconnaissance within scope

What requires explicit authorisation:
- Any reconnaissance against a client's systems (written scope of work)
- Using breach data to attempt authentication against live systems
- Social engineering employees
- Physical reconnaissance of premises

Jurisdiction matters:
- GDPR (EU) — personal data of EU residents has protections even when publicly available
- CFAA (US) — accessing systems without authorisation, even for OSINT, can violate the act
- UK Computer Misuse Act — similar to CFAA

Always have written authorisation before starting any recon against a real target.

In authorised engagements, document every finding with source and timestamp. This protects both you and your client if findings become part of legal proceedings or regulatory review.
