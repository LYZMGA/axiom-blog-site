Red vs Blue: The Eternal Game

2026-06-16 · 1 min read · Security · red-team blue-team purple-team attack-defense mitre-attack detection adversary-simulation

The cat-and-mouse between attackers and defenders has no finish line. A structured breakdown of every major kill-chain phase — attacker technique on the left, defender response on the right — and why purple teaming is the only way to close the loop.

Every intrusion is a conversation. The attacker sends a move; the defender responds or misses it. The game has been running since the first worm crawled a network in 1988 and it will never end — because the asymmetry is fundamental: attackers need to find one gap, defenders must close every gap simultaneously.

This post maps the kill chain phase by phase. For each stage: what the attacker does, what artefacts they leave, and what a competent defender should have in place to detect or deny it. Read both columns. Build the right one. Test with the left.

01 The Asymmetry Problem

Before the phase-by-phase breakdown, the structural reality defenders must accept:

  • Attackers operate in time — they can spend weeks on reconnaissance before touching a target system
  • Defenders operate in real-time — they must detect while managing change, patching, and business operations simultaneously
  • Attackers can iterate freely — a failed technique costs them time; it costs a defender nothing because they may not have known about it
  • Defenders bear the burden of proof — every alert must be investigated; attackers pay no cost for false positives in your SIEM

None of this means defence is futile. It means the game is not symmetric and your strategy must account for that.

02 Phase 1 — Reconnaissance

ATTACKERDEFENDER
**Passive OSINT**: LinkedIn employee harvesting, job postings reveal tech stack, GitHub leaks internal hostnames and API keys, WHOIS history shows acquired domains**Reduce exposure**: audit GitHub org for secrets, scrub job postings of internal tool names, enforce email format privacy on public sites
**Active recon**: Shodan/Censys enumerate exposed services, certificate transparency reveals internal subdomains, DNS brute-force maps infrastructure**Monitor certificates**: subscribe to crt.sh alerts for your domains. Shodan Monitor for your IP ranges. Run your own Shodan searches monthly
**Tools**: theHarvester, maltego, recon-ng, amass, subfinder — entirely passive, never touches target systems**Control the attack surface**: internet exposure review quarterly. Ask: should this port be open? Should this service be internet-facing at all?
Defender advantage in phase 1

Reconnaissance is the phase where defenders have the strongest hand — because the attacker is gathering public data, you can audit the same sources they use. Run the attacker's tools against yourself once a quarter. Every internal hostname, credential, or technology detail you find in public sources is one the adversary already has.

03 Phase 2 — Initial Access

ATTACKERDEFENDER
**Spearphishing**: crafted email referencing legitimate context from OSINT, malicious attachment or credential-harvesting link, targets with VPN or email access**Email security stack**: SPF + DKIM + DMARC enforcement (reject policy), sandboxed attachment detonation, link rewriting and real-time reputation checks
**Exploit public-facing app**: VPN gateway auth bypass, unpatched web app RCE, exposed admin panels with default/weak credentials**Patch SLA enforcement**: critical internet-facing systems get 24–48 hour SLA. Run Shodan against your own IP ranges weekly. Web app scanning in CI/CD
**Supply chain / trusted software**: trojanised update, compromised developer dependency, malicious VS Code extension**Software composition analysis**: enforce dependency signing, vendor security questionnaires, monitor update channels for anomalies
**Valid credentials**: purchased from initial access brokers (stolen from previous breach), password spraying on OWA/VPN, credential stuffing from combo lists**MFA everywhere**: FIDO2/passkeys on VPN and email make credential theft worthless. Monitor for password spray patterns (many failures across few accounts)
bash 
# Red team: test your own perimeter for initial access vectors (authorised)

# Find exposed admin interfaces
nmap -p 8080,8443,9090,4443,3389,5900,5985 --open <your-ranges>

# Password spray test (authorised, with IR team aware)
# Use low-and-slow: 1 attempt per account per 30 minutes stays under lockout threshold
python3 spray.py --user-list users.txt --password 'Winter2026!' --target mail.corp.com --interval 30

# Check your domains against breach databases
curl -s 'https://haveibeenpwned.com/api/v3/breachedaccount/user@corp.com' \
  -H 'hibp-api-key: YOUR_KEY'

# Blue team: detect password spray in Wazuh / SIEM
# Many unique usernames, same source IP, short time window, all fail (event 4625)
# Rule: >20 unique usernames with 4625 from same IP in 10 minutes = spray

04 Phase 3 — Execution

ATTACKERDEFENDER
**PowerShell (T1059.001)**: encoded commands, download-cradles (`IEX (New-Object Net.WebClient).DownloadString(...)`), bypass execution policy**Script block logging + AMSI**: enable PowerShell v5 ScriptBlock Logging (Event 4104), Constrained Language Mode, log and alert on encoded `-enc` flags
**WMI (T1047)**: `wmic process call create`, wmiprvse.exe spawns attacker's binary — difficult to attribute in many SIEMs**WMI event subscription hunting**: Sysmon Event ID 19/20/21 captures WMI subscriptions. Alert on wmiprvse.exe spawning shells or suspicious children
**Macro-enabled documents (T1137)**: Office macro downloads and executes stager, MSHTA, or rundll32 LOLBin chain**Disable macros by policy**: GPO to block macros from internet-origin documents. Defender Attack Surface Reduction rules block Office child processes
**Living-off-the-land binaries**: certutil, mshta, rundll32, regsvr32, bitsadmin — signed Microsoft binaries abused to download/execute**LOLBin detection**: alert on certutil -urlcache, mshta.exe with http:// arguments, regsvr32 with /s /u /i:http. These are almost always malicious in modern environments

05 Phase 4 — Persistence

ATTACKERDEFENDER
**Registry Run keys (T1547.001)**: `HKCU\Software\Microsoft\Windows\CurrentVersion\Run` — survives user logon, low privilege required**Autoruns monitoring**: Sysinternals Autoruns comparison on gold image vs live system. Wazuh file integrity monitoring (FIM) on registry hives
**Scheduled tasks (T1053.005)**: `schtasks /create`, often with SYSTEM context, hidden in `\Microsoft\Windows\` folder**Task baseline and alerting**: Sysmon EID 1 on schtasks.exe, Windows Event 4698 (task created). Alert on tasks created outside change windows
**Services (T1543.003)**: malicious service installed, runs on boot as SYSTEM, blends with legitimate service names**Service integrity monitoring**: baseline all services on gold image. Alert on new service installations (Event 7045). Check service binary paths for anomalies
**WMI Event Subscriptions (T1546.003)**: fileless persistence, survives reboots, notoriously hard to detect without dedicated tooling**Hunt WMI subscriptions**: `Get-WMIObject -Namespace root\subscription -Class __EventFilter`. PowerShell script to dump and diff all subscriptions weekly
powershell 
# Blue team: hunt for persistence mechanisms (run on Windows endpoints)

# Registry Run key inspection
$runKeys = @(
    'HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\Run',
    'HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\RunOnce',
    'HKCU:\SOFTWARE\Microsoft\Windows\CurrentVersion\Run'
)
foreach ($key in $runKeys) {
    Get-ItemProperty -Path $key -ErrorAction SilentlyContinue | 
        Select-Object * -ExcludeProperty PSPath,PSParentPath,PSChildName,PSProvider
}

# Scheduled tasks — find non-Microsoft tasks
Get-ScheduledTask | Where-Object { $_.TaskPath -notmatch 'Microsoft' } |
    Select-Object TaskName, TaskPath, State, @{n='Command';e={$_.Actions.Execute}}

# Services with non-standard binary paths
Get-WmiObject Win32_Service |
    Where-Object { $_.PathName -notmatch 'System32|SysWOW64|Program Files' } |
    Select-Object Name, StartMode, State, PathName

# WMI Event subscriptions (fileless persistence)
Get-WMIObject -Namespace root\subscription -Class __EventFilter
Get-WMIObject -Namespace root\subscription -Class __EventConsumer
Get-WMIObject -Namespace root\subscription -Class __FilterToConsumerBinding

06 Phase 5 — Privilege Escalation

ATTACKERDEFENDER
**Token impersonation (T1134)**: SeImpersonatePrivilege abuse via Potato exploits (Hot/Sweet/Rogue) — nearly guaranteed SYSTEM from IIS/SQL service accounts**Remove unnecessary privileges**: service accounts should not have SeImpersonatePrivilege unless required. Audit with `whoami /priv` on each service account context
**Unquoted service paths (T1574.009)**: `C:\Program Files\My App\service.exe` → attacker plants `C:\Program.exe`**Audit unquoted paths**: `wmic service get name,pathname | findstr /v '"' | findstr /v 'C:\Windows'`. Fix with quotes in service registry key
**AlwaysInstallElevated (T1548.002)**: MSI files install as SYSTEM when this GPO is set — trivial privilege escalation**Disable via GPO**: Computer Configuration > Policies > Administrative Templates > Windows Components > Windows Installer > Always install with elevated privileges = Disabled
**Kerberoasting (T1558.003)**: request Kerberos service tickets for SPNs, crack offline — no special privilege needed, entirely in-memory**Protect service accounts**: use Group Managed Service Accounts (gMSA) with 240-char random passwords. Alert on 4769 events with encryption type 0x17 (RC4) at scale

07 Phase 6 — Lateral Movement

ATTACKERDEFENDER
**Pass-the-Hash (T1550.002)**: NTLM hash from memory used directly — no password cracking needed, works for remote services**Enable Protected Users group**: prevents NTLM for members. Credential Guard on Windows 10+ isolates LSA secrets in hypervisor-protected memory
**PsExec / SMB lateral (T1021.002)**: copies binary to ADMIN$, creates service — classic but still works in environments without EDR coverage**Alert on ADMIN$ writes**: Sysmon network connections + service creation from remote hosts. Firewall rule: restrict SMB port 445 to management hosts only
**WinRM / PowerShell Remoting (T1021.006)**: `Enter-PSSession`, `Invoke-Command` — built-in, encrypted, legitimate — blends perfectly**Restrict WinRM**: allow only from jump hosts via firewall. Log all remote PowerShell sessions (Event 4103/4104). Alert on first-time WinRM connections between workstations
**DCOM (T1021.003)**: `MMC20.Application`, `ShellWindows` — used by APT groups because it leaves minimal artefacts and uses legitimate COM infrastructure**DCOM hunting**: Sysmon network connections from svchost.exe or mmc.exe to remote hosts. Baseline which systems legitimately use DCOM

08 Phase 7 — Impact / Objectives

ATTACKERDEFENDER
**Data exfiltration (T1048)**: HTTPS to cloud services (OneDrive, Dropbox, Mega) — encrypted and whitelisted; DNS exfiltration via `data.attacker.com` TXT records**DNS monitoring**: flag unusually long subdomains (>50 chars), high-frequency queries to new domains, queries to domains registered <30 days ago
**Ransomware deployment (T1486)**: RaaS affiliate deploys encryptor after weeks of dwell time — encryption is the last step, not the breach**Canary files**: plant fake documents in file shares with unique names. Any read = immediate alert. Also monitor for mass file rename events (ransomware extension pattern)
**Domain persistence (T1207, T1484)**: DCSync, Golden/Silver tickets, AdminSDHolder modification — attackers in AD maintain access long after initial cleanup**Privileged Access Workstations**: DCSync generates Event 4662 on the DC for the DS-Replication-Get-Changes right. Alert on any non-DC account triggering this

09 The Purple Team Bridge

Red team finds gaps. Blue team builds detections. Purple team closes the loop between them. Without structured purple teaming, red team reports become expensive documents that sit in a backlog.

The purple team model is simple: red and blue work the same attack chain together, in real time. Red executes technique T1059.001. Blue checks if the SIEM fired. If not, blue writes the detection rule. Red tests again. Iterate until the technique is covered, then move to the next one.

This is fundamentally different from a traditional pentest where the red team hands over a PDF and leaves.

bash 
# Purple team workflow — Atomic Red Team + SIEM validation

# Step 1: Red team executes a specific ATT&CK technique
Invoke-AtomicTest T1003.001 -TestNumbers 1  # LSASS memory dump via Task Manager

# Step 2: Blue team queries SIEM for the expected artefact
# In Wazuh/Kibana:
# query: rule.mitre.technique:"T1003.001" AND timestamp:[now-5m TO now]
# Expected: Wazuh rule 92211 or Sysmon EID 10 on lsass.exe

# Step 3: If no alert — write the detection rule and re-test
# Step 4: Document in coverage matrix (ATT&CK Navigator layer)

# Generate ATT&CK Navigator layer showing detection coverage
python3 - << 'EOF'
import json

# Techniques with confirmed detection
covered = ['T1059.001', 'T1003.001', 'T1547.001', 'T1558.003', 'T1021.002']
no_coverage = ['T1021.003', 'T1207', 'T1048', 'T1134']

layer = {
    'name': 'SOC Lab Coverage',
    'domain': 'enterprise-attack',
    'techniques': [
        {'techniqueID': t, 'color': '#4caf50', 'comment': 'Detected'} for t in covered
    ] + [
        {'techniqueID': t, 'color': '#f44336', 'comment': 'No detection'} for t in no_coverage
    ]
}
print(json.dumps(layer, indent=2))
EOF
# Upload output to https://mitre-attack.github.io/attack-navigator/
The uncomfortable truth

Most organisations detect fewer than 15% of the ATT&CK techniques an average red team uses. The MITRE ATT&CK Evaluations consistently show that leading EDR products miss technique variants that differ slightly from their trained patterns. The answer is not a better product — it is a structured programme of attack simulation and detection validation repeated continuously. Purple teaming is not a project. It is a practice.
