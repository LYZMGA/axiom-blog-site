Networking & Subnetting — The Complete Foundation

2026-06-04 · 9 min read · Learn · networking subnetting tcp-ip osi red-team fundamentals

OSI model, TCP/IP stack, IPv4 addressing, CIDR subnetting, protocols, ARP, routing, NAT, VLANs, and a red team lens on every layer.

Networking is the invisible foundation underneath every engagement, CTF, and piece of infrastructure. You cannot do meaningful reconnaissance, exploitation, or lateral movement without understanding how data actually travels between machines.

This guide builds from zero — OSI layers, IP addresses, subnets — up through the concepts that directly matter in practice: reading a CIDR range, understanding a TCP handshake, spotting a misconfigured network. Red team notes are woven throughout, not bolted on at the end.

01 The OSI Model

The OSI (Open Systems Interconnection) model is a conceptual framework that divides network communication into seven layers. Every real protocol lives at one or more of these layers, and understanding the stack is essential for knowing where something can go wrong — or be attacked.

LayerNameWhat it doesExamples
7**Application**User-facing protocols, data formatsHTTP, DNS, FTP, SMTP, SSH
6**Presentation**Encoding, encryption, compressionTLS/SSL, JPEG, ASCII
5**Session**Managing sessions between hostsSMB sessions, NetBIOS, RPC
4**Transport**End-to-end delivery, portsTCP, UDP
3**Network**Logical addressing and routingIP, ICMP, ARP
2**Data Link**Physical addressing on a local segmentEthernet, MAC addresses, VLANs
1**Physical**Raw bits over a mediumCables, radio, fiber

Mnemonic: Please Do Not Throw Sausage Pizza Away (layers 1→7).

Why it matters for attackers:
- Layer 2 attacks: ARP poisoning, MAC flooding, VLAN hopping
- Layer 3 attacks: IP spoofing, ICMP redirects, route injection
- Layer 4 attacks: TCP SYN floods, port scanning, session hijacking
- Layer 7 attacks: SQLi, XSS, SSRF, every web vulnerability you know

When you run nmap, you're operating at layers 3–4. When you intercept with Burp Suite, you're at layer 7. Knowing which layer you're touching changes what tools you use and what defenses exist.

02 The TCP/IP Model

The TCP/IP model (also called the Internet model) is the practical implementation — four layers that map to the OSI seven. This is what real operating systems implement.

TCP/IP LayerMaps to OSIProtocols
**Application**5, 6, 7HTTP, HTTPS, DNS, FTP, SSH, SMTP, SNMP
**Transport**4TCP, UDP
**Internet**3IP, ICMP, ARP
**Network Access**1, 2Ethernet, Wi-Fi, PPP

Data travels down the stack on the sender (encapsulation) and up on the receiver (decapsulation). Each layer wraps the data from the layer above with its own header:

text 
Application data
  → [TCP header] + data          (segment)
  → [IP header] + segment         (packet)
  → [Ethernet header] + packet    (frame)
  → bits on the wire

This is called encapsulation. When you capture with Wireshark you're reading these headers at every layer simultaneously.

03 IPv4 Addressing

Every device on a network needs a unique logical address — the IP address. IPv4 uses 32 bits, written as four octets (groups of 8 bits) in dotted decimal.

text 
192.168.1.100
│   │   │ │
│   │   │ └─ 4th octet  (0–255)
│   │   └─── 3rd octet  (0–255)
│   └─────── 2nd octet  (0–255)
└─────────── 1st octet  (0–255)

In binary:

text 
192     .168     .1       .100
11000000.10101000.00000001.01100100

Every IPv4 address has two parts: the network portion and the host portion. The subnet mask defines where the split is.

Address Classes (historical, mostly replaced by CIDR)

ClassRangeDefault MaskUsage
A1.0.0.0 – 126.255.255.255/8Large networks
B128.0.0.0 – 191.255.255.255/16Medium networks
C192.0.0.0 – 223.255.255.255/24Small networks
D224.0.0.0 – 239.255.255.255—Multicast
E240.0.0.0 – 255.255.255.255—Reserved/Experimental

Private (RFC 1918) Ranges

These are non-routable on the public internet — you'll see them constantly on internal networks and in CTFs:

RangeCIDRTypical use
10.0.0.0 – 10.255.255.25510.0.0.0/8Large internal networks, VPNs
172.16.0.0 – 172.31.255.255172.16.0.0/12Docker, VMs, corporate
192.168.0.0 – 192.168.255.255192.168.0.0/16Home/small office networks

Special addresses:
- 127.0.0.1 — loopback (localhost)
- 169.254.x.x — APIPA (link-local, no DHCP response)
- 0.0.0.0 — unspecified / wildcard listener
- 255.255.255.255 — limited broadcast

04 Subnet Masks & CIDR Notation

A subnet mask is a 32-bit number where all network bits are 1 and all host bits are 0. It tells you which part of an IP address is the network address.

CIDR (Classless Inter-Domain Routing) notation writes the mask as a prefix length — the count of leading 1 bits.

CIDRSubnet MaskBinary
/8255.0.0.011111111.00000000.00000000.00000000
/16255.255.0.011111111.11111111.00000000.00000000
/24255.255.255.011111111.11111111.11111111.00000000
/25255.255.255.12811111111.11111111.11111111.10000000
/26255.255.255.19211111111.11111111.11111111.11000000
/27255.255.255.22411111111.11111111.11111111.11100000
/28255.255.255.24011111111.11111111.11111111.11110000
/29255.255.255.24811111111.11111111.11111111.11111000
/30255.255.255.25211111111.11111111.11111111.11111100
/32255.255.255.25511111111.11111111.11111111.11111111

/30 is common for point-to-point links (2 usable hosts). /32 means a single host — you'll see this in routing tables and firewall rules.

05 Subnet Math — Step by Step

Given 192.168.10.75/26, find the network address, broadcast, usable range, and number of hosts.

Step 1 — Find the block size
The last octet mask bits: /26 means 26 bits total, the last octet contributes 26 - 24 = 2 bits. So the mask in the last octet is 11000000 = 192. Block size = 256 - 192 = 64.

Step 2 — Find the network address
Divide the host octet (75) by the block size (64): 75 / 64 = 1 remainder 11. The network starts at 1 × 64 = 64.
→ Network: 192.168.10.64

Step 3 — Broadcast address
Network address + block size - 1: 64 + 64 - 1 = 127
→ Broadcast: 192.168.10.127

Step 4 — Usable host range
First usable: network + 1 → 192.168.10.65
Last usable: broadcast - 1 → 192.168.10.126

Step 5 — Number of hosts
Host bits = 32 - 26 = 6. Hosts = 2^6 - 2 = 62 usable.

Summary for .75/26:

Value
Network192.168.10.64
Broadcast192.168.10.127
First host192.168.10.65
Last host192.168.10.126
Usable hosts62

    print(sub)
"

# Check if an IP is in a subnet
python3 -c "import ipaddress; print(ipaddress.ip_address('10.10.10.50') in ipaddress.ip_network('10.10.10.0/24'))"

06 TCP — Transmission Control Protocol

TCP is connection-oriented, reliable, and ordered. It guarantees that data arrives in the right order and retransmits lost segments. Used by HTTP, HTTPS, SSH, FTP, SMTP — anything where reliability matters.

The Three-Way Handshake

Before any data is exchanged, TCP establishes a connection:

text 
Client                          Server
  │──── SYN (seq=x) ──────────────▶│   "I want to connect"
  │◀─── SYN-ACK (seq=y, ack=x+1) ─│   "OK, and I'm ready"
  │──── ACK (ack=y+1) ────────────▶│   "Acknowledged"
  │                                 │
  │══════ data flows both ways ═════│

TCP Flags

FlagMeaningAttacker use
**SYN**Synchronize / initiatePort scanning, SYN flood DoS
**ACK**Acknowledge received dataACK scan (pass stateless firewalls)
**FIN**Finish, close connection gracefullyFIN scan, evasion
**RST**Reset, abort connectionRST injection, TCP hijack detection
**PSH**Push data immediately to application—
**URG**Urgent pointer is validRarely used
**NULL**No flags setNULL scan (RFC 793 evasion)
**XMAS**FIN+PSH+URG setXMAS scan (RFC 793 evasion)

TCP Connection Teardown

text 
Client                          Server
  │──── FIN ──────────────────────▶│
  │◀─── ACK ───────────────────────│
  │◀─── FIN ───────────────────────│
  │──── ACK ──────────────────────▶│

RST immediately terminates without the four-way close. Firewalls inject RSTs to kill disallowed connections. Nmap uses RST responses to detect closed ports.

07 UDP — User Datagram Protocol

UDP is connectionless and unreliable — fire and forget. No handshake, no retransmit, no ordering guarantee. Much faster overhead than TCP. Used by DNS (port 53), DHCP (67/68), SNMP (161), NTP (123), and any application that tolerates loss (video streams, VoIP, games).

From an attacker perspective:
- UDP services are slower to scan (no RST on closed ports — just ICMP unreachable or silence)
- UDP often has less visibility in logs
- SNMP on UDP 161 is frequently misconfigured with default community strings (public, private)
- DNS on UDP 53 can be abused for amplification DDoS and tunneling
- TFTP (UDP 69) sometimes left open on network devices — no authentication

08 ICMP — Internet Control Message Protocol

ICMP is the network's diagnostic protocol — it carries error messages and operational information. It has no ports; it uses type and code numbers instead.

TypeMeaningCommon use
0Echo Replyping response
3Destination Unreachableport/host not reachable
8Echo Requestping
11Time ExceededTTL expired — used by `traceroute`
5RedirectRouting redirect (can be abused)

Red team uses of ICMP:
- ping for host discovery when TCP/UDP is filtered
- traceroute / tracert maps network hops to reveal routers and segmentation
- ICMP tunneling (ptunnel, icmptunnel) — exfiltrate data or C2 through firewalls that allow ICMP
- ICMP type 3 code 3 (port unreachable) confirms a host is up but port is closed

bash ICMP / ping / traceroute 
# Basic ping
ping -c 4 10.10.10.1

# Ping sweep a /24 (bash)
for i in $(seq 1 254); do
  ping -c1 -W1 192.168.1.$i &>/dev/null && echo "192.168.1.$i UP" &
done; wait

# fping — fast concurrent ping sweep
fping -ag 10.10.10.0/24 2>/dev/null

# Traceroute (Linux)
traceroute 8.8.8.8
traceroute -T -p 80 8.8.8.8   # TCP SYN traceroute (bypasses ICMP blocks)

# MTR — live traceroute
mtr --report 8.8.8.8

# ICMP tunnel (data exfiltration channel)
# Server side
ptunnel-ng -v 5
# Client side
ptunnel-ng -p <server_ip> -lp 8080 -da <destination> -dp 22

10 ARP — Address Resolution Protocol

When a device knows the IP address of the next hop but needs the MAC address to actually send a frame on Ethernet, it uses ARP.

text 
Host A wants to reach 192.168.1.5

A → (broadcast) "Who has 192.168.1.5? Tell 192.168.1.1"
B ← (unicast)   "192.168.1.5 is at aa:bb:cc:dd:ee:ff"

A caches: 192.168.1.5 → aa:bb:cc:dd:ee:ff  (ARP table / ARP cache)

ARP has no authentication. Any host can reply to any ARP request — or send gratuitous ARPs unprompted to overwrite entries.

ARP Poisoning (ARP Spoofing)

Send forged ARP replies to two targets, telling each that you are the other:

text 
Attacker tells Victim:  "The gateway (192.168.1.1) is at MY MAC"
Attacker tells Gateway: "Victim (192.168.1.100) is at MY MAC"

Result: all traffic routes through attacker → Man-in-the-Middle

Tools: arpspoof (dsniff), Ettercap, Bettercap, scapy.

bash ARP enumeration and attack 
# View your ARP cache
arp -n
ip neigh

# ARP scan — discover hosts on local segment (faster than ping sweep)
arp-scan --localnet
arp-scan 192.168.1.0/24 -I eth0

# Nmap ARP discovery (requires root)
nmap -sn 192.168.1.0/24 --send-ip

# ARP poisoning with arpspoof (enable IP forwarding first!)
echo 1 > /proc/sys/net/ipv4/ip_forward
arpspoof -i eth0 -t 192.168.1.100 192.168.1.1   # tell victim attacker = gateway
arpspoof -i eth0 -t 192.168.1.1 192.168.1.100   # tell gateway attacker = victim

# Bettercap — ARP spoof + sniff
bettercap -iface eth0
# In bettercap:
# net.probe on
# set arp.spoof.targets 192.168.1.100
# arp.spoof on
# net.sniff on

11 Routing & the Default Gateway

A router connects networks. When a packet's destination is outside the local subnet, the sender forwards it to the default gateway — the router's IP on the local segment.

How a host decides where to send a packet

  1. Is the destination in the same subnet? → Send directly (ARP for MAC, then Ethernet frame)
  2. Not in the same subnet? → Send to default gateway (router handles it from there)

The Routing Table

Every device (host or router) has a routing table. Routers match the most specific (longest prefix) route.

text 
Destination      Gateway       Mask        Interface
0.0.0.0          192.168.1.1   0.0.0.0     eth0     ← default route
192.168.1.0      0.0.0.0       /24         eth0     ← local subnet (direct)
10.0.0.0         192.168.1.254 /8          eth0     ← static route to 10.x.x.x
bash Routing table commands 
# Linux — view routing table
ip route show
route -n

# Windows
route print
ipconfig /all         # includes default gateway

# Add a static route (Linux)
ip route add 10.10.0.0/16 via 192.168.1.254 dev eth0

# Add a static route (Windows)
route add 10.10.0.0 mask 255.255.0.0 192.168.1.254

# Trace the path a packet takes
traceroute 8.8.8.8

# Red team: read routes on a compromised host to map internal segments
# Found 10.0.0.0/8 via a pivot? That's your lateral movement target

12 NAT — Network Address Translation

NAT allows many devices with private (RFC 1918) addresses to share a single public IP. The router rewrites source IPs and tracks state in a NAT table.

text 
Internal:  192.168.1.50:4231  ──→  NAT router  ──→  Public: 203.0.113.1:4231
                                    (rewrites src IP)
Response:  203.0.113.1:4231  ──→  NAT router  ──→  192.168.1.50:4231
                                    (looks up state, rewrites dst)

Types of NAT:
- SNAT (Source NAT / Masquerade) — outbound traffic; your home router does this
- DNAT (Destination NAT / Port Forwarding) — inbound, maps external port to internal host
- PAT (Port Address Translation) — many-to-one, uses port numbers to track flows

Red team implications:
- Hosts behind NAT are not directly reachable from the internet — you need a pivot or DNAT entry
- Port forwarding rules expose internal services: find these in router configs
- NAT can hide internal network topology, but routes and ARP tables on a pivot reveal it
- Double NAT (NAT inside NAT) is common in cloud environments and VMs

13 VLANs — Virtual Local Area Networks

VLANs segment a physical network into isolated logical networks at Layer 2. Devices on VLAN 10 cannot directly communicate with devices on VLAN 20 without a router (or Layer 3 switch) — even if they share the same physical switch.

802.1Q Tagging — switches insert a 4-byte tag into the Ethernet frame header containing the VLAN ID (1–4094).

Trunk ports carry multiple VLANs (tagged). Access ports belong to a single VLAN (untagged to the device).

VLAN Hopping

Two main techniques to jump between VLANs without going through a router:

1. Switch Spoofing — trick the switch into forming a trunk link with your machine by sending DTP (Dynamic Trunking Protocol) frames. Once trunking, you send/receive tagged frames for any VLAN.

2. Double Tagging — embed two 802.1Q tags. The outer tag matches the native VLAN (stripped by the first switch). The inner tag targets the victim VLAN (forwarded by the second switch). Only works one-way.

Mitigations: Disable DTP (switchport nonegotiate), set native VLAN to an unused ID, disable unused ports, use port security.

bash VLAN enumeration and hopping 
# Create a VLAN interface on Linux (for testing/pivoting)
modprobe 8021q
ip link add link eth0 name eth0.20 type vlan id 20
ip addr add 10.20.0.100/24 dev eth0.20
ip link set eth0.20 up

# Enumerate VLANs via CDP/LLDP on a trunk port
yersinia -G   # GUI — sends DTP, CDP, LLDP frames

# VLAN hopping with yersinia (switch spoofing attack)
yersinia dtp -attack 1 -interface eth0

# Scapy — manual double-tag packet
python3 -c "
from scapy.all import *
pkt = Ether()/Dot1Q(vlan=1)/Dot1Q(vlan=20)/IP(dst='10.20.0.1')/ICMP()
sendp(pkt, iface='eth0')
"

# After gaining trunk access — scan a target VLAN
nmap -e eth0.20 -sn 10.20.0.0/24

14 IPv6 Basics

IPv6 uses 128-bit addresses written as eight groups of four hex digits separated by colons. Double colon :: compresses consecutive all-zero groups.

text 
2001:0db8:0000:0000:0000:ff00:0042:8329
→ 2001:db8::ff00:42:8329  (compressed)

Important IPv6 address types:

AddressRangeMeaning
Loopback`::1`localhost
Link-local`fe80::/10`Auto-configured, local segment only
Global unicast`2000::/3`Publicly routable
Multicast`ff00::/8`One-to-many
`::ffff:x.x.x.x`—IPv4-mapped IPv6

Link-local addresses (fe80::) are automatically assigned to every IPv6-capable interface using the EUI-64 algorithm (derived from the MAC address). This is significant: you can often determine a device's MAC from its link-local address.

Red team notes:
- IPv6 is often overlooked in firewalls and monitoring — separate rulesets required
- IPv6 is enabled by default on modern Windows/Linux even on IPv4-only networks
- NDP (Neighbour Discovery Protocol) replaces ARP — same poisoning attacks apply (radvd, THC-IPv6, fake RA messages)
- Tools: ping6, nmap -6, thc-ipv6 suite

bash IPv6 enumeration 
# View IPv6 addresses
ip -6 addr show

# Ping6
ping6 -c 3 ::1
ping6 -c 3 fe80::1%eth0   # link-local requires interface scope

# Nmap IPv6 scan
nmap -6 -sV fe80::a00:27ff:fe4e:66a1%eth0
nmap -6 -sn 2001:db8::/32

# Discover neighbours (replaces ARP)
ip -6 neigh show

# Router Advertisement flood (fake IPv6 router — causes client misconfiguration)
thc-ipv6 fake_router6 eth0 2001:db8::/64

# Rogue RA — redirect IPv6 traffic through attacker
# (can bypass IPv4-only firewall rules entirely)
fake_router26 eth0

15 DHCP — Dynamic Host Configuration Protocol

DHCP automatically assigns IP addresses to hosts on a network. Four-step process (DORA):

text 
Client  ──── DHCPDISCOVER (broadcast) ────────▶  (all servers)
Server  ◀─── DHCPOFFER ──────────────────────────
Client  ──── DHCPREQUEST (I'll take that offer) ▶
Server  ◀─── DHCPACK (confirmed) ────────────────

DHCP also delivers: subnet mask, default gateway, DNS servers, domain name, NTP servers, WINS servers.

Red team attacks:
- Rogue DHCP — spin up a fake DHCP server faster than the real one. Set yourself as the default gateway or DNS server → MitM all traffic on the segment
- DHCP starvation — flood with DHCPDISCOVER using spoofed MACs, exhaust the IP pool, then bring up rogue DHCP
- Info gathering — DHCP ACK packets leak internal domain name, gateway, DNS — capture with Wireshark on a new connection

16 Red Team Network Recon Workflow

Put it all together. When you land on a network (fresh shell, new VPN, physical access), this is the mental model:

  1. Where am I? — Get your IP, subnet, gateway, DNS
  2. What's on this segment? — ARP scan, ping sweep, passive listening
  3. What segments exist? — Read the routing table, check for VLANs
  4. What's reachable? — Port scan the live hosts
  5. What's interesting? — Identify services, grab banners, check for defaults
bash Full local network recon workflow 
# 1. Where am I?
ip addr show
ip route show
cat /etc/resolv.conf
hostname -I

# Windows equivalent
ipconfig /all
route print

# 2. Passive — listen for traffic before making noise
tcpdump -i eth0 -nn -c 200 2>/dev/null   # see what's broadcasting/talking

# 3. Host discovery — ARP (LAN) then ICMP (routed)
arp-scan --localnet
nmap -sn 192.168.1.0/24                  # ping scan
nmap -sn 192.168.1.0/24 -PE -PP -PS21,22,80,443  # aggressive discovery

# 4. Quick port scan on live hosts
nmap -T4 -F 192.168.1.0/24              # fast top-100 ports
nmap -T4 --open -p- 192.168.1.10       # all ports on single host

# 5. Service/version scan on interesting hosts
nmap -sV -sC -p 22,80,443,445,3389 192.168.1.10

# 6. Banner grabbing
nc -nv 192.168.1.10 80
openssl s_client -connect 192.168.1.10:443 2>/dev/null | head -20

# 7. Check for common misconfigs
nmap --script smb-vuln* -p 445 192.168.1.0/24
nmap --script ftp-anon -p 21 192.168.1.0/24
nmap --script snmp-info -sU -p 161 192.168.1.0/24

17 Wireshark — Reading the Wire

Wireshark is the standard packet analyser. Understanding packet captures ties together everything above — you see the actual handshakes, ARP requests, DNS queries, and plaintext credentials in real time.

Essential display filters:

FilterWhat it shows
`ip.addr == 192.168.1.50`All traffic to/from that IP
`tcp.port == 80`HTTP traffic
`dns`All DNS queries and responses
`arp`ARP requests and replies
`icmp`All ICMP (ping, traceroute, unreachables)
`tcp.flags.syn == 1 && tcp.flags.ack == 0`SYN packets (new connections / scans)
`http.request.method == "POST"`Form submissions, login attempts
`ftp`FTP commands (credentials in clear)
`smtp`Email in transit
`!(arp or dns or icmp)`Hide noise, show real traffic

Follow a stream: Right-click a packet → Follow → TCP Stream. Reconstructs the full conversation as text — essential for reading HTTP sessions, FTP commands, or anything plaintext.

bash tcpdump — command-line packet capture 
# Capture all traffic on eth0
tcpdump -i eth0

# Capture with full packet content (hex + ASCII)
tcpdump -i eth0 -XX

# Filter by host
tcpdump -i eth0 host 192.168.1.50

# Filter by port
tcpdump -i eth0 port 80
tcpdump -i eth0 port 53

# Capture SYN packets only (new connections)
tcpdump -i eth0 'tcp[tcpflags] & tcp-syn != 0'

# Save to file, open in Wireshark later
tcpdump -i eth0 -w capture.pcap

# Read a pcap
tcpdump -r capture.pcap

# Strip DNS/ARP noise and show only real traffic
tcpdump -i eth0 not arp and not port 53

# Capture credentials from FTP / Telnet / HTTP
tcpdump -i eth0 -A -s0 port 21 or port 23 or port 80 | grep -iE 'user|pass|login|password'

18 Key Takeaways

Concepts to have cold:

  • OSI layers 1–7 and where each attack class lives
  • IPv4 binary → decimal conversion and subnet math without a calculator
  • CIDR prefix → number of hosts: /24=254, /25=126, /26=62, /30=2
  • Private ranges: 10/8, 172.16/12, 192.168/16
  • TCP three-way handshake and what each flag means
  • ARP is unauthenticated — always a MitM vector on a local segment
  • Default gateway is the escape hatch from the subnet — compromise it and you see everything
  • VLANs provide logical segmentation but can be hopped if DTP is enabled or native VLAN is misconfigured
  • IPv6 is enabled by default and often unmonitored — a parallel network to audit separately

Tools to know:
ipcalc, nmap, arp-scan, wireshark, tcpdump, bettercap, ip, traceroute, scapy

Networking fundamentals don't expire. The TCP/IP stack that existed in 1981 still underlies every HTTP request, every reverse shell, every LDAP query. Building a solid mental model here makes every layer above it faster to learn and easier to break.
